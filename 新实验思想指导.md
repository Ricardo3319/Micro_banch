# RescueSched: Problem, Model, and Pseudo-code 定稿版

## 1. Problem Statement

Modern latency-critical RPC services often run on commodity multi-core servers with one worker pinned to each CPU core. To avoid centralized queue contention and preserve cache locality, each worker maintains a local request queue. Incoming packets are commonly distributed by RSS, flow hashing, or a user-space ingress thread. This design scales well under balanced load, but it can create transient per-core queue imbalance under bursty arrivals, skewed flow distribution, and heterogeneous service times.

For microsecond-scale RPC services, even a short period of local queue buildup can cause P99 or P99.9 tail-latency SLO violations. Existing work stealing improves work conservation by allowing idle cores to pull tasks from busy queues. Work shedding and threshold-based migration react when a queue becomes overloaded. However, these mechanisms mainly answer the question:

> Which core has too much work?

They do not explicitly answer three finer-grained SLO questions:

1. Which queued request will miss its SLO if it stays local?
2. Which of those requests can still meet its SLO after migration?
3. Which target core can accept the migrated request without becoming risky itself?

This paper studies the following problem:

> Given a commodity multi-core RPC server with per-core request queues and no preemption of running requests, can we reduce SLO violation ratio by selectively migrating only queued-but-not-running requests that are locally doomed, remotely rescuable, and safe for the target core?

The key distinction is that RescueSched is not a generic load balancer. It does not aim to equalize queue lengths. Instead, it treats cross-core migration as a scarce and costly SLO-rescue action. A migration is useful only if it changes the SLO outcome of a request without creating new target-side violations.

## 2. Scope and Non-goals

### 2.1 Target Setting

We target a single latency-critical RPC service running on one multi-core server. The service has:

* (N) pinned worker cores;
* one FIFO queue per worker core;
* short request service times, typically in the microsecond range;
* request-level SLOs or deadlines;
* no preemption of running requests;
* no Linux kernel modification;
* no reliance on SmartNIC, programmable switch, or special CPU interrupt features.

The scheduler only migrates request descriptors that are still queued. It does not migrate already-running requests, application state, payload buffers, or network packets already being processed.

### 2.2 Non-goals

This work does not attempt to replace preemptive schedulers, single-queue dispatchers, or cross-application core allocation systems. It also does not attempt to solve global overload. If all cores are saturated and no target core has slack, RescueSched should suppress migration rather than move requests around uselessly.

## 3. System Model

### 3.1 Cores and Queues

Let the server have a set of worker cores:

\mathcal{C} = {1,2,\ldots,N}.

Each core (c \in \mathcal{C}) maintains a FIFO queue:

Q_c(t)=\langle r_1,r_2,\ldots,r_{|Q_c|}\rangle.

At time (t), core (c) may also be executing one request. Let:

R_c(t)

denote the estimated remaining processing time of the currently running request on core (c). If the core is idle, then:

R_c(t)=0.

The estimated queued workload on core (c) is:

W_c(t)=R_c(t)+\sum_{r_i \in Q_c(t)} \hat{s}_i.

Here, (\hat{s}_i) is the estimated service time of request (r_i).

## 4. Request Model

Each request is represented as:

r_i = \langle id_i, a_i, D_i, m_i, \hat{s}_i, c_i, state_i \rangle.

Where:

| Symbol      | Meaning                                 |
| ----------- | --------------------------------------- |
| (id_i)      | request identifier                      |
| (a_i)       | arrival timestamp                       |
| (D_i)       | deadline                                |
| (m_i)       | method or request class                 |
| (\hat{s}_i) | estimated service time                  |
| (c_i)       | current queue/core                      |
| (state_i)   | queued, running, completed, or migrated |

The deadline is:

D_i = a_i + SLO_i.

For a fixed-SLO service, (SLO_i = SLO). For class-aware services, (SLO_i) can depend on method (m_i).

The scheduler only considers requests with:

state_i = queued.

Running requests are never migrated.

## 5. Service-time Estimation

The scheduler maintains lightweight per-method service-time statistics. For each method or class (m), it maintains:

\mu_m(t)

as an EWMA estimate of service time:

\mu_m(t) = (1-\alpha)\mu_m(t-1) + \alpha s_i,

where (s_i) is the observed service time of a completed request of method (m), and (\alpha) is a smoothing factor.

The estimated service time of request (r_i) is:

\hat{s}_i = \mu_{m_i}(t).

For heavy-tailed workloads, a conservative estimate can be used:

\hat{s}_i = Q_p(S_{m_i}),

where (Q_p(S_{m_i})) is the empirical (p)-th percentile service time for method (m_i), such as (p=0.90) or (p=0.95).

The minimal implementation should start with EWMA mean. Quantile-aware estimation can be used as an ablation.

## 6. Local Completion-time Estimate

For a queued request (r_i) currently in source core (s), define the set of requests ahead of it in the local queue as:

Before(i,s)={r_k \in Q_s(t) \mid r_k \text{ is ahead of } r_i}.

The estimated local completion time is:

\hat{C}_{local}(i,s)
=
t + R_s(t) + \sum_{r_k \in Before(i,s)} \hat{s}_k + \hat{s}_i.

Request (r_i) is locally doomed if:

\hat{C}_{local}(i,s) > D_i.

A locally doomed request is expected to violate its SLO if it remains in its current queue.

## 7. Migration Cost Model

For a candidate migration from source core (s) to target core (j), define the estimated migration cost as:

M_{s,j}(i)
=
M_{desc} + M_{sync} + M_{numa}(s,j).

Where:

| Term            | Meaning                                    |
| --------------- | ------------------------------------------ |
| (M_{desc})      | descriptor movement cost                   |
| (M_{sync})      | queue synchronization or notification cost |
| (M_{numa}(s,j)) | additional NUMA penalty                    |

In the first implementation, migration is restricted to cores within the same NUMA domain, so:

M_{numa}(s,j)=0.

Thus:

M_{s,j}(i) \approx M_{desc}+M_{sync}.

The value of (M_{s,j}(i)) should be calibrated by microbenchmarking descriptor-only cross-core enqueue/dequeue latency.

## 8. Remote Completion-time Estimate

If request (r_i) is migrated from source core (s) to target core (j), it will be appended to the tail of (Q_j(t)). The estimated remote completion time is:

\hat{C}_{remote}(i,j)
=
t + M_{s,j}(i) + R_j(t) + \sum_{r_k \in Q_j(t)} \hat{s}_k + \hat{s}_i.

The request is remotely feasible on target (j) if:

\hat{C}_{remote}(i,j) + \epsilon \le D_i.

Here, (\epsilon) is a safety margin used to absorb service-time estimation error and migration-cost uncertainty.

## 9. Rescuable Request Definition

A request (r_i) is rescuable from source core (s) to target core (j) if and only if:

\hat{C}_{local}(i,s) > D_i

and:

\hat{C}_{remote}(i,j) + \epsilon \le D_i.

Equivalently:

Rescuable(i,s,j)=
\mathbb{I}
[
\hat{C}_{local}(i,s) > D_i
\land
\hat{C}_{remote}(i,j)+\epsilon \le D_i
].

This definition intentionally excludes two bad cases:

1. Needless migration: the request would have met its SLO locally.
2. Unsaved migration: the request would still miss its SLO after migration.

The remaining remote slack is:

Slack_{remote}(i,j)=D_i-\hat{C}_{remote}(i,j).

A larger (Slack_{remote}) means the migration is more robust to estimation error.

## 10. Target-side Risk Model

A target core should not be selected only because its queue is short. A short queue can still be risky if the queued requests have tight deadlines or long service times.

Define the predicted number of SLO misses in queue (Q_j(t)) as:

Risk(j,t)
=
\sum_{r_x \in Q_j(t)}
\mathbb{I}
[
\hat{C}_{local}(x,j) > D_x
].

If request (r_i) is appended to (Q_j(t)), the resulting queue is:

Q_j^{+i}(t)=Q_j(t)\oplus r_i.

The post-migration risk is:

Risk_{after}(j|i,t)
=
\sum_{r_x \in Q_j^{+i}(t)}
\mathbb{I}
[
\hat{C}_{after}(x,j|i) > D_x
].

The incremental target-side risk is:

\Delta Risk(j|i,t)
=
Risk_{after}(j|i,t)-Risk(j,t).

A target core (j) is migration-safe for request (r_i) if:

\Delta Risk(j|i,t) \le \theta.

The default strict setting is:

\theta=0.

This means RescueSched does not allow a migration that is predicted to create additional target-side SLO violations.

## 11. Migration Usefulness Taxonomy

Each migration is classified after completion. This classification is used for evaluation and optional feedback control.

### 11.1 Beneficial Migration

A migration is beneficial if:

C_{local}^{cf}(i)>D_i

and:

C_{actual}(i)\le D_i.

Here, (C_{local}^{cf}(i)) is the counterfactual completion time if the request had stayed local, and (C_{actual}(i)) is the actual completion time after migration.

### 11.2 Needless Migration

A migration is needless if:

C_{local}^{cf}(i)\le D_i.

This means the request would have met its SLO locally, so migration was unnecessary.

### 11.3 Unsaved Migration

A migration is unsaved if:

C_{local}^{cf}(i)>D_i

and:

C_{actual}(i)>D_i.

This means the request was indeed at risk, but migration failed to rescue it.

### 11.4 Harmful Migration

A migration is harmful if it causes one or more requests already queued at the target core to miss their SLO:

Harmful(i,j)=
\mathbb{I}
[
ActualMisses(Q_j^{+i}) > ActualMisses(Q_j)
].

### 11.5 Useless Migration Ratio

The useless migration ratio is:

UMR =
\frac{
N_{needless}+N_{unsaved}+N_{harmful}
}{
N_{migrated}
}.

The beneficial migration ratio is:

BMR =
\frac{
N_{beneficial}
}{
N_{migrated}
}.

The rescue-per-migration metric is:

RPM =
\frac{
N_{rescued}
}{
N_{migrated}
}.

These metrics show whether the scheduler migrates precisely, not merely frequently.

## 12. Optimization Objective

At each scheduling epoch (t), RescueSched selects a set of migration pairs:

\mathcal{X}(t)={(i,j)}.

Each pair means that request (r_i) is migrated to target core (j).

The objective is:

\max_{\mathcal{X}(t)}
\sum_{(i,j)\in \mathcal{X}(t)}
[
Gain(i,j)
-
\lambda \Delta Risk(j|i,t)
-
\beta M_{s,j}(i)
].

Where:

Gain(i,j)=Rescuable(i,s,j).

The constraints are:

\sum_j x_{i,j} \le 1,

meaning each request can be migrated at most once per epoch.

\sum_{i,j} x_{i,j} \le B,

meaning total migrations per epoch are bounded by migration budget (B).

\Delta Risk(j|i,t) \le \theta,

meaning target safety must hold.

state_i = queued,

meaning running requests are excluded.

In implementation, this optimization is solved by a lightweight greedy algorithm over a bounded candidate window. No heavy online optimizer is required.

## 13. Core Design Insight

The main insight is:

> For microsecond-scale RPC services, migration is only valuable when it changes the SLO outcome of a queued request. Queue length alone identifies overloaded cores, but it does not identify which task is still worth rescuing or which target core can absorb it safely.

Therefore, RescueSched uses queue length only as a risk trigger, not as the final migration decision.

The final migration decision depends on:

1. local completion-time counterfactual;
2. remote completion-time counterfactual;
3. migration cost;
4. remaining deadline slack;
5. incremental target-side risk.

## 14. Algorithm 1: RescueSched Main Loop

Algorithm 1: RescueSched Main Scheduling Loop

Input:
  C: set of worker cores
  Q_c: local queue of each core c
  B: migration budget per scheduling epoch
  K: maximum number of candidate requests scanned per risky queue
  H: maximum number of candidate target cores
  epsilon: safety margin for remote completion
  theta: maximum allowed incremental target risk
  lambda, beta: scoring weights

On every scheduling epoch t or queue update event:

1. Update per-core queue statistics:
     for each core c in C:
         update R_c(t)
         update W_c(t)
         update Risk(c,t)

2. Identify risky source cores:
     S_risky = { s in C | Risk(s,t) > 0 or W_s(t) exceeds risk trigger }

3. Identify low-risk target cores:
     T_safe = { j in C | Risk(j,t) = 0 or W_j(t) is below low-risk trigger }

4. Initialize candidate set:
     Pairs = empty set

5. For each source core s in S_risky:
       Candidates = select_top_K_risk_window(Q_s, K)

       For each request i in Candidates:
           if state_i != queued:
               continue

           C_local = estimate_local_completion(i, s)

           if C_local <= D_i:
               continue
               // needless migration: skip

           For each target core j in select_target_candidates(T_safe, H):
               if j == s:
                   continue

               C_remote = estimate_remote_completion(i, j)

               if C_remote + epsilon > D_i:
                   continue
                   // unsaved migration: skip

               delta = estimate_delta_risk(j, i)

               if delta > theta:
                   continue
                   // unsafe target: skip

               score = compute_score(i, j, C_local, C_remote, delta)

               Add (i, j, score) to Pairs

6. Sort Pairs by descending score.

7. Selected = empty set

8. For each pair (i, j, score) in Pairs:
       if |Selected| >= B:
           break

       if request i has already been selected:
           continue

       if target j is no longer safe:
           continue

       Add (i, j) to Selected
       update simulated target state for j

9. Execute descriptor-only migrations:
       for each (i, j) in Selected:
           atomically remove i from its source queue
           append i to target queue j
           mark i as migrated
           log migration decision

10. Return.

## 15. Algorithm 2: Rescuable Candidate Selection

Algorithm 2: select_top_K_risk_window(Q_s, K)

Input:
  Q_s: source queue
  K: scan window size

Output:
  Candidate requests likely to be locally doomed

1. Candidates = empty list
2. cumulative_wait = R_s(t)

3. For each request i in Q_s from head to tail:
       C_local_i = t + cumulative_wait + estimated_service(i)

       if C_local_i > D_i:
           Add i to Candidates

       cumulative_wait += estimated_service(i)

       if |Candidates| == K:
           break

4. Return Candidates.

This scan does not simply take the tail of the queue. It selects requests whose predicted local completion time exceeds their deadline. This is the first boundary between RescueSched and threshold-based tail migration.

## 16. Algorithm 3: Migration-safe Target Selection

Algorithm 3: estimate_delta_risk(j, i)

Input:
  j: candidate target core
  i: candidate migrated request

Output:
  delta risk caused by appending request i to Q_j

1. risk_before = 0
2. cumulative_wait = R_j(t)

3. For each request x in Q_j from head to tail:
       C_x = t + cumulative_wait + estimated_service(x)

       if C_x > D_x:
           risk_before += 1

       cumulative_wait += estimated_service(x)

4. risk_after = 0
5. cumulative_wait = R_j(t)

6. For each request x in Q_j followed by i:
       C_x_after = t + cumulative_wait + estimated_service(x)

       if C_x_after > D_x:
           risk_after += 1

       cumulative_wait += estimated_service(x)

7. delta = risk_after - risk_before

8. Return delta.

In the minimal version, (i) is appended to the target queue tail. This preserves FIFO order in the target queue. More aggressive insertion policies should not be included in the first version.

## 17. Algorithm 4: Migration Scoring

Algorithm 4: compute_score(i, j, C_local, C_remote, delta)

Input:
  i: candidate request
  j: target core
  C_local: predicted local completion time
  C_remote: predicted remote completion time
  delta: incremental target risk

Output:
  migration score

1. local_lateness = max(0, C_local - D_i)
2. remote_slack = D_i - C_remote
3. mig_cost = estimate_migration_cost(source(i), j)

4. if local_lateness <= 0:
       return -infinity

5. if remote_slack < epsilon:
       return -infinity

6. if delta > theta:
       return -infinity

7. score =
       remote_slack
       + gamma * local_lateness
       - lambda * delta
       - beta * mig_cost

8. return score.

A simple first version can set:

\gamma=0,
\quad
\theta=0.

Then the score becomes:

Score(i,j)=Slack_{remote}(i,j)-\beta M_{s,j}(i).

This keeps the mechanism lightweight and easier to debug.

## 18. Complexity

Let:

| Symbol | Meaning                                              |
| ------ | ---------------------------------------------------- |
| (N)    | number of worker cores                               |
| (K)    | number of candidate requests scanned per risky queue |
| (H)    | number of candidate target cores                     |
| (L)    | average queue length scanned for target risk         |

The naive per-epoch complexity is:

O(|S_{risky}| \cdot K \cdot H \cdot L).

To keep overhead low, the implementation should:

1. scan only the first (K) locally doomed candidates;
2. consider only the top (H) low-risk target cores;
3. maintain per-core prefix sums of estimated service time;
4. maintain approximate risk summaries for target queues.

With prefix sums, local completion estimates can be computed in:

O(1)

per request after queue snapshotting. Target risk estimation can be approximated in:

O(KH)

per epoch when only the incremental effect on a bounded queue window is evaluated.

## 19. Minimal Prototype Policy

For the first implementation, use the following conservative policy:

1. Only same-NUMA migration.
2. Only queued request descriptors are migrated.
3. No running request preemption.
4. No payload copy.
5. Append migrated request to target tail.
6. (\theta=0), meaning no predicted additional target misses.
7. (B) is fixed per scheduling epoch.
8. Candidate scan window (K) starts from 8 or 16.
9. Target candidate count (H) starts from 2 or 4.
10. Service time estimate starts with EWMA mean.

This version is sufficient to test the core claim.

## 20. Baselines Required by the Model

The model must be evaluated against the following baselines:

1. RSS-local FIFO / d-FCFS:

   * each core processes only its own queue.

2. Work stealing:

   * idle cores pull work from non-empty queues.

3. Work shedding:

   * overloaded cores push work to less-loaded cores.

4. ALTO-style threshold migration:

   * when queue length exceeds a threshold, tail requests are migrated to lightly loaded queues.

5. Shortest-queue migration:

   * candidate requests are migrated to the core with the shortest queue.

6. Random-target migration:

   * candidate requests are migrated to randomly selected low-load cores.

The most important comparisons are:

* RescueSched vs work stealing;
* RescueSched vs work shedding;
* RescueSched vs ALTO-style threshold migration;
* safe target vs shortest queue;
* rescuable selection vs tail migration.

## 22. Core Claim to Test

The core empirical claim should be:

> RescueSched reduces SLO violation ratio not by migrating more requests, but by increasing the fraction of migrations that are actually beneficial.

This claim is stronger than simply showing lower P99 latency. It directly supports the paper’s novelty.

## 23. Expected Main Result Pattern

The expected result should be:

1. Under balanced homogeneous workloads:

   * RescueSched performs similarly to work stealing.
   * It performs few migrations.
   * This shows it does not overreact.

2. Under bursty and skewed workloads:

   * RescueSched reduces SLO violation ratio compared with local FIFO and work stealing.
   * It outperforms threshold migration by avoiding needless and unsaved migrations.

3. Under mixed short/long or heavy-tailed workloads:

   * RescueSched benefits from task-level service-time estimation.
   * Safe target selection reduces harmful target-side misses.

4. Under global overload:

   * RescueSched suppresses migration.
   * It should not claim to solve overload without available slack.

## 24. Final Paper-level Contribution Statement

The paper should claim exactly three contributions:

1. We define task-level rescuability for microsecond-scale RPC scheduling, using local-vs-remote completion-time counterfactuals under migration cost.

2. We design migration-safe target selection, which prevents cross-core migration from transferring SLO risk from the source core to the target core.

3. We implement RescueSched as a commodity user-space prototype and show that selective descriptor migration reduces SLO violations and useless migrations compared with work stealing, work shedding, and threshold-based migration baselines.

## 25. Final Pseudo-code Summary

RescueSched in one sentence:

When a core becomes risky, scan only queued requests that are predicted
to miss locally; for each such request, test whether any low-risk target
can complete it before its deadline without increasing target-side risk;
then migrate only the highest-score descriptor pairs within a small budget.

This is the final algorithmic identity of the work.
