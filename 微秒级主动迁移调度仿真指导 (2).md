# **数据中心微秒级主动迁移调度离散事件仿真（DES）项目指导文档**

在当今全面向微服务架构演进的云数据中心中，应用间的远程过程调用（RPC）耗时已逐渐缩短至微秒甚至纳秒级别。传统基于操作系统内核网络协议栈的调度方案，因其固有的上下文切换、中断处理以及锁竞争开销，已无法满足现代交互式应用对极低尾延迟（Tail Latency）的服务等级目标（SLO）。尽管以 DPDK 和 eRPC 为代表的内核旁路（Kernel-bypass）技术大幅度剥离了网络 I/O 的绝对时间，将端到端网络延迟压缩至个位数微秒 1，但数据中心网络和计算层面的新瓶颈随之浮现。随着异构计算资源的普及（如降频节点、多代 CPU 混部）与瞬时突发流量（Micro-bursts）的频发，排队延迟（Queueing Delay）超越了处理延迟，成为决定微秒级 RPC SLO 是否违约的核心因素 1。

现有的微秒级调度系统通常可以划分为两类：一是基于被动工作窃取（Work-stealing）的节点内调度系统（如 ZygOS 1）；二是基于近似最短队列（Join-Shortest-Queue, JSQ）或 Power-of-d-Choices 随机分发的机架级调度系统（如 RackSched 1）。然而，这些方案在应对异构环境时存在致命缺陷：被动工作窃取只有在核心空闲时才会触发，对于已经陷入慢节点（Straggler）的微秒级任务而言，等待被窃取往往意味着 SLO 已经被打破；而基于分发层的策略一旦做出初次分配，便无法纠正因局部流量突增导致的排队积压。

本研究的核心创新在于提出一种**主动式、感知通信开销的分布感知队列批迁移算法**。在任务违反 SLO 前，系统不再对核心等待队列中的每一个任务逐一打分和排序，而是根据服务时间分布（如双峰分布、重尾分布）构建核心队列的紧凑摘要，利用排队论估计一批任务在本地继续等待时的违约风险。若某个队列批次的风险超过阈值，且陈旧全局视图中存在更优的远端节点，系统将主动把这一批仍处于等待队列中的 RPC 请求元数据迁移到目标主机，以缓解 Head-of-Line 阻塞和慢节点积压。

为了在 INFOCOM 等顶尖学术会议上提供可辩护的实验支撑，必须构建一个高保真度的离散事件仿真（Discrete Event Simulation, DES）器，并用小规模实机实验校准关键物理常数。最终实机验证平台确定为 **CloudLab c6525-25g**：根据 CloudLab 官方硬件说明（https://docs.cloudlab.us/hardware.html），该类型包含 AMD EPYC 7302P 16-core 3.00GHz CPU、128GB ECC 内存、两块 480GB SATA SSD、两块 dual-port Mellanox ConnectX-5 25Gb NIC，并提供两条 25Gb 实验链路。论文与实验文档应以 `c6525-25g` 为准，不再使用泛化的“c6525 128核/100GbE”表述。仿真中的 $T\_{host}$、$T\_{net}$、队列摘要构建成本和请求转发成本应在该平台上通过 microbenchmark 校准；在校准前，`T_net_oneway_us=3.15` 与 `T_host_us=2.1` 只能作为初始假设，而不是最终实测结论。

## **零、顶层研究定位与当前算法路线（规划版）**

本节用于固定当前项目的最高层设计方向。它不是代码实现规范，而是论文选题、算法边界和实验闭环的统一口径。后续代码修改、实验脚本和论文写作均应围绕本节展开。

### **0.1 核心研究问题**

本项目的核心研究问题定义为：

> 在微秒级 RPC 系统中，当任务已经被初始分发到某个主机/核心队列后，如何在远端负载视图陈旧、控制面预算极小、服务时间分布高离散的条件下，主动识别即将造成尾延迟违约的队列形态，并以低开销方式迁移一批仍处于等待状态的请求，从而降低 P99/P99.9 尾延迟？

这个问题的重点不是重新设计初始分发器，也不是迁移正在执行的线程或内存状态，而是研究 **post-dispatch queue repair**：当 Power-of-k、JSQ 或静态分发已经把任务放入局部队列后，系统是否还能在 SLO 被打破之前，以极小比例的跨主机请求转发修复“坏队列”。

### **0.2 论文级主张**

最终论文不应被表述为“实现了一个主动迁移启发式算法”，而应表述为：

> 微秒级 RPC 系统中的尾延迟风险往往由核心队列的服务时间分布形态决定，而不是由某个孤立任务的局部分数决定。因此，主动迁移控制面不应逐任务打分排序，而应构建固定大小的队列分布摘要，识别双峰/重尾负载下的批次级风险模式，并对有限数量的等待请求执行有界批迁移。

该主张对应的关键词为：

- `Distribution-aware`：利用双峰、重尾等服务时间分布信息。
- `Queue-batch`：以核心队列中的任务批次为决策对象。
- `Bounded overhead`：控制路径复杂度与队列长度解耦。
- `Stale-view robust`：只使用陈旧全局视图和本机实时队列状态。
- `Calibrated simulation`：大规模仿真参数必须由 c6525-25g 小规模实测校准。

### **0.3 与现有算法阶段的关系**

当前项目中已有或计划保留的算法角色如下：

| 方法 | 论文角色 | 是否作为最终创新 |
| :---- | :---- | :---- |
| `B1 Power-of-k` | 初始分发 baseline | 否 |
| `B2 ReactiveMigration` | 阈值触发被动迁移 baseline | 否 |
| `M0 ProactiveMigration` | 逐任务预测迁移 baseline，用于证明主动机制有价值 | 否 |
| `M1 AQB-PM` | 当前批量选择原型，用于暴露逐任务候选排序的局限 | 否，最多作为过渡版本 |
| `DQB-PM` | 分布感知队列批迁移，最终目标算法 | 是 |

因此，后续论文主线应从 `M0/M1` 转向 `DQB-PM`。`M0` 的价值是提供逐任务预测迁移基线；`M1` 的价值是说明批量决策有潜力但仍存在逐任务候选排序、局部 reservation、固定前缀扫描等问题；`DQB-PM` 才是面向 INFOCOM 投稿的目标算法。

### **0.4 DQB-PM 的顶层设计原则**

1. **不逐任务全量排序**  
   控制面不得依赖对队列中所有任务逐一打分再排序。允许维护固定大小分布桶，允许扫描固定前缀或维护候选指针，但主控制路径必须保持有界复杂度。

2. **用分布摘要估计队列风险**  
   双峰负载下维护 `short/long` 两类摘要；重尾负载下维护 `mice/medium/elephant` 摘要。每个桶至少记录任务数量、累计工作量、最老任务等待时间、最紧 SLO slack 和可迁移候选指针。

3. **用排队论估计批次违约风险**  
   对于某一任务批，风险估计由当前 running task 剩余时间、队列前缀工作量、桶内平均服务时间、核心容量和任务已等待时间共同决定。迁移触发条件不再是单个任务的 `local_total_i > alpha * SLO_i`，而是批次级风险质量 `risk_mass` 或批次尾部估计延迟超过阈值。

4. **识别负载分布特定模式**  
   双峰分布重点识别 `ShortBehindLong`：长任务阻塞一批短任务。重尾分布重点识别 `MiceBehindElephant`：少数 elephant task 造成大量小任务尾延迟恶化。异构场景重点识别 `SlowNodeBatchPressure`：慢节点核心上工作量积压导致一批任务共同逼近 SLO。

5. **目标选择必须考虑 incoming reservation**  
   远端目标选择不能只看陈旧 workload。必须维护短窗口 `incoming_reservation`，把已调度但尚未到达的迁移工作量计入目标负载，避免多个源主机同时涌入同一个看似空闲的目标。

6. **全局饱和时主动少迁移或不迁移**  
   如果集群整体 P25/P50 workload 已经很高，迁移只是在搬移排队延迟。此时算法应提高 margin、降低 batch cap，必要时进入 no-migrate 模式。

7. **迁移语义必须克制**  
   本项目迁移的是等待队列中的 RPC 请求元数据或 queued request descriptor，不迁移正在执行的任务、线程栈、内存页或部分执行状态。该边界是论文可信度的基础。

### **0.5 预期创新点与非目标**

**真正的核心创新**：

- 分布感知的核心队列摘要。
- 队列批次级 SLO 风险估计。
- stale-view 下带 incoming reservation 的批迁移目标选择。
- 控制面复杂度与队列长度解耦。

**重要但属于实现层的贡献**：

- C++ 离散事件仿真器。
- O(1) 队列抽离数据结构。
- histogram 指标统计。
- CloudLab c6525-25g microbenchmark 校准。

**当前阶段不作为主线目标**：

- 完整 P4/SmartNIC 实现。
- 完整 kernel-bypass RPC runtime。
- 迁移正在执行任务的运行状态。
- 复杂 RL/ML 调度器。
- packet-level 全数据中心网络仿真。

### **0.6 证据闭环**

最终论文应形成三层证据：

1. **机制证据**：通过消融实验说明分布摘要、批迁移、incoming reservation、饱和保护分别解决什么问题。
2. **规模证据**：通过大规模 DES 在 64-node 及更大规模下验证 P99/P99.9、SLO 违约率、迁移率和无效迁移率。
3. **现实证据**：通过 CloudLab c6525-25g 小规模实测校准网络延迟、主机处理开销、队列摘要构建成本、请求转发成本，并证明小规模趋势与仿真一致。

### **0.7 前沿研究吸收与取舍**

本项目需要吸收近年微秒级 RPC、数据中心调度和高速调度论文中的方法论，但不能盲目复刻已有系统。以下取舍作为当前研究路线的边界。

| 代表工作 | 可吸收内容 | 不直接采用的内容 | 对本项目的启发 |
| :---- | :---- | :---- | :---- |
| [Horus, NSDI 2024](https://www.usenix.org/conference/nsdi24/presentation/yassini) | 微秒级任务调度必须低状态、低通信开销，并要在动态负载下优化 tail response time。 | 不实现 in-network scheduler，不依赖 programmable switch。 | 我们聚焦 post-dispatch queue repair，用主机侧低开销摘要补足初始调度之后的纠偏能力。 |
| [RackSched, OSDI 2020](https://www.usenix.org/conference/osdi20/presentation/zhu) | 两层调度、rack-scale 视角、微秒级 SLO 需要网络-系统协同。 | 不构建 ToR 交换机调度器，不要求 12-server/Tofino 级原型。 | 论文需说明 DQB-PM 与初始 rack-level dispatch 不冲突，而是修复已落地队列。 |
| [Shinjuku, NSDI 2019](https://www.usenix.org/conference/nsdi19/presentation/kaffes) | 高离散/重尾服务时间会造成短任务被长任务阻塞，HoL 是核心机制。 | 不实现微秒级抢占 OS，不把本地抢占作为主贡献。 | DQB-PM 的 `ShortBehindLong` 和 `MiceBehindElephant` 模式必须用实验拆开证明。 |
| [Concord, SOSP 2023](https://dslab.epfl.ch/research/concord/) | 近似最优、低开销、可部署，比严格最优更适合微秒级系统。 | 不追求全局最优调度证明。 | DQB-PM 应强调 bounded-overhead approximate queue repair，而不是 oracle 最优迁移。 |
| [R2P2, ATC 2019](https://www.usenix.org/conference/atc19/presentation/kogias-r2p2) | RPC 应作为网络一等公民，目标选择和请求传输可以分离。 | 不实现新的 RPC transport 或 P4 middlebox。 | 本项目迁移语义应定义为 queued RPC request forwarding，而不是线程/进程迁移。 |
| [SIRD, NSDI 2025](https://www.usenix.org/conference/nsdi25/presentation/prasopoulos) | 单边调度会在共享链路或共享目标处冲突，需要 sender/receiver 信息协调。 | 不设计新的拥塞控制协议。 | DQB-PM 的目标选择必须加入 `incoming_reservation`，避免 cross-source herding。 |
| 高速近似调度/PIFO 类工作 | 高速系统中全量排序代价过高，常用 bucket、近似排序或有限候选。 | 不实现硬件 packet scheduler。 | 支持“不逐任务排序”的论文论点，队列摘要和固定桶是合理的系统取舍。 |
| CloudLab 官方硬件文档 | 小规模实机平台必须明确，仿真常数要可校准。 | 不声称 c6525-25g 能复现大规模绝对数值。 | c6525-25g 用于校准和趋势验证，而不是替代大规模仿真。 |

由此得到的最终取舍是：**保留低开销、分布感知、stale-view 鲁棒和小规模校准这些思想；放弃 P4/SmartNIC、完整 RPC transport、微秒级抢占 OS 和 RL 调度器这些高风险路线。**

### **0.8 完整研究路径**

当前项目按五个阶段推进：

**阶段 A：问题定义与模型建立**

- 明确研究对象为微秒级 RPC 的等待队列请求，而不是正在执行任务。
- 明确系统只允许读本机实时队列状态和远端陈旧视图。
- 建立 workload 模型：双峰、重尾、MMPP burst、异构慢节点和目标端 incoming pressure。
- 建立成本模型：网络单程、主机处理、队列摘要维护、控制周期、请求转发。

**阶段 B：DQB-PM 方法设计**

- `QueueSummary`：每个核心维护固定数量分布桶。
- `PatternDetector`：识别 `ShortBehindLong`、`MiceBehindElephant`、`SlowNodeBatchPressure`。
- `BatchRiskEstimator`：用排队论估计批次级 SLO 风险。
- `DestinationSelector`：基于 stale workload、目标容量和 incoming reservation 选目标。
- `BatchCommitter`：在迁移预算、batch cap、per-dst cap 下提交一批等待请求。
- `GuardController`：在全局饱和、目标拥塞或 invalid ratio 升高时主动降级。

**阶段 C：大规模仿真**

- 主平台为当前自研 C++ DES，而不是 ns-3 packet-level 仿真。
- 主规模为 64 hosts x 16 cores，并补充 128/256 hosts 扩展性扫描。
- 主指标为 P99/P99.9、SLO violation、migration rate、invalid migration ratio、control overhead、short/long 分组 SLO。
- 必须比较 `B1/B2/M0/M1/DQB-PM`，并加入 local priority 或 workload-aware dispatch 作为补充 baseline。

**阶段 D：CloudLab c6525-25g 小规模校准**

- 使用 2-4 台 c6525-25g 测 RPC RTT、请求转发成本、批元数据转发成本、队列摘要更新成本和控制周期成本。
- 用实测参数回灌仿真，检查主结论是否保持。
- 小规模验证目标是趋势一致和参数可信，不要求绝对值复现大规模仿真。

**阶段 E：论文闭环**

- Introduction 的 tension：微秒级任务量巨大，逐任务排序不可行；现有分发/窃取/本地抢占不能修复所有坏队列。
- Design 的核心：分布摘要、批次风险、incoming reservation、饱和保护。
- Evaluation 的主线：机制有效、规模有效、副作用可控、实机校准可信。
- Discussion 必须主动承认边界：不迁移执行状态、不处理所有应用类型、不保证全局最优。

### **0.9 DQB-PM 的算法对象与数据流**

后续代码实现应围绕以下逻辑数据流展开：

```text
Core wait queue
  -> QueueSummary(short/long or mice/medium/elephant)
  -> PatternDetector
  -> QueueBatchCandidate
  -> BatchRiskEstimator
  -> DestinationSelector(stale_workload + incoming_reservation)
  -> BatchCommitter
  -> Metrics/Diagnostics
```

算法只在 `QueueBatchCandidate` 层面做收益判断，不在全队列任务层面做全量排序。真正迁移时，`BatchCommitter` 迁移的是某个分布区域或微批次的 FIFO 前缀，迁移规模由任务数上限、工作量上限、目标端可接收 slack 和全局迁移预算共同决定。单个 task 仍然是执行和统计单位，但不是控制面的决策单位。若未来加入真实 batch transport，再把同一 `(src_host, dst_host)` 的多个任务聚合成一个 `MigrationBatch` 事件；但这不是 MVP 的必要条件。

### **0.10 当前算法完整流程（DQB-PM）**

本节从一次请求生命周期和一次迁移控制周期两个角度，完整描述当前目标算法 DQB-PM 的执行流程。该流程是顶层设计说明，后续代码实现应尽量保持接口与语义一致。

#### **0.10.1 初始化阶段**

仿真或系统启动时，首先初始化集群拓扑、服务时间模型、SLO 分层、陈旧全局视图和控制面参数：

1. **集群初始化**：创建 `N` 个 host，每个 host 包含 `C` 个 core。每个 core 有独立等待队列、当前运行任务指针、容量系数 `C_j` 和队列摘要结构 `QueueSummary`。
2. **异构容量初始化**：同构实验中所有 core 的 `C_j=1.0`；异构实验中部分 host/core 的 `C_j` 降低，如 `0.2`，用于模拟慢节点、降频节点或 noisy neighbor。
3. **服务时间分布初始化**：
   - 双峰负载：维护 `short/long` 两类，短任务如 `E<=20us`，长任务如 `E>20us`。
   - 重尾负载：维护 `mice/medium/elephant` 三类，可按服务时间阈值或分位数划分。
4. **SLO 初始化**：根据任务服务时间类别设置 SLO，例如短任务 `SLO_short=40us`，长任务 `SLO_long=200us`。
5. **控制面初始化**：设置 `SYNC_LOAD` 周期、`CHECK_MIGRATION` 周期、迁移预算、batch cap、per-destination cap、cooldown、margin、saturation guard 和 `incoming_reservation` 窗口。

#### **0.10.2 请求生成与初始分发**

每个请求先经历普通分发路径，DQB-PM 不替代入口分发器：

1. `TASK_GENERATE` 根据 workload 生成一个新请求，记录 `generate_time_us`、真实服务时间 `base_service_time_us` 和估计服务时间 `expected_service_time_us`。
2. 调度器使用 baseline-compatible 的初始分发策略，例如 Power-of-k、workload-aware dispatch 或其他入口策略，选择目标 host。
3. 系统为该请求生成一个 `TASK_ARRIVE` 事件，时间戳为 `now + T_net_oneway`。
4. 该阶段只决定请求第一次落在哪个 host，不做主动迁移判断。

#### **0.10.3 请求到达与核心队列更新**

请求到达目标 host 后，host-local dispatcher 选择一个 core：

1. 若目标 core 空闲，请求直接进入执行，生成 `TASK_FINISH` 事件。
2. 若目标 core 忙碌，请求插入该 core 的等待队列尾部。
3. 插入等待队列时，同步更新该 core 的 `QueueSummary`：
   - 按 `expected_service_time_us` 把任务归入 `short/long` 或 `mice/medium/elephant` 桶。
   - 更新桶内任务数、累计工作量、最老任务等待时间、最紧 SLO slack。
   - 可维护每个桶的候选指针或固定大小候选窗口，用于后续批迁移时快速取出任务。

该阶段是 DQB-PM 与普通调度器的第一个区别：队列不是只记录长度，还记录服务时间分布和风险摘要。

#### **0.10.4 状态同步与陈旧全局视图**

系统周期性触发 `SYNC_LOAD`：

1. 每个 host 汇总本地队列状态，包括总队列长度、总 workload、按容量归一化后的 workload，以及可选的分布摘要。
2. 全局视图 `stale_view` 和 `stale_workload_view` 被更新。
3. DQB-PM 在迁移决策时只能读取这份陈旧视图，不允许读取远端实时队列。
4. 已经调度但尚未到达目标 host 的迁移任务不会立即反映在 `stale_workload_view` 中，因此必须额外维护 `incoming_reservation[dst]`。

这一步保证算法满足分布式系统约束：本机状态实时可见，远端状态只能延迟可见。

#### **0.10.5 迁移检查触发**

每个 host 周期性触发 `CHECK_MIGRATION`，也可以在队列压力突增时触发。一次检查流程如下：

1. 检查 host-level cooldown。若该 host 刚刚发生迁移，则跳过本轮。
2. 检查全局迁移预算。若迁移率已接近预算上限，则跳过或降低 batch cap。
3. 检查 saturation guard。若全局 P25/P50 workload 已过高，说明迁移可能只是搬移排队延迟，此时提高 margin、降低 batch cap，严重时进入 no-migrate 模式。
4. 对该 host 的每个 core 读取本地 `QueueSummary`，准备生成批候选。

#### **0.10.6 队列模式识别**

DQB-PM 不逐任务全量打分，而是先判断每个 core 当前属于哪种风险模式：

1. **`ShortBehindLong`**  
   双峰负载下，若 running task 或队列前部存在长任务，并且后方短任务数量、累计等待时间或最老短任务 age 已接近 `alpha * SLO_short`，则认为存在长任务阻塞短任务批的风险。

2. **`MiceBehindElephant`**  
   重尾负载下，若 elephant task 占据大量 workload，且 mice 任务桶中的最老任务或风险质量升高，则认为小任务批正在被超长任务拖入尾延迟。

3. **`SlowNodeBatchPressure`**  
   异构场景下，若慢节点核心上累计 workload 经容量归一化后显著偏高，并且一批任务的本地预计完成时间接近 SLO，则生成慢节点批迁移候选。

4. **`GenericPressure`**  
   若没有明显分布模式，但本地 workload 和队列 age 已显示批量违约风险，则生成通用压力候选。

每个 core 每个控制周期最多生成固定数量的 `QueueBatchCandidate`，通常为 0 或 1 个，以保证控制复杂度有界。

#### **0.10.7 批次级本地风险估计**

对每个 `QueueBatchCandidate`，估计如果不迁移，这一批任务继续留在本地队列中的尾部风险：

1. 计算 running task 剩余时间：

   $$R = \max(0, finish\_time - now)$$

2. 计算候选批前方的预计等待工作量 `W_before`，包括：
   - running task 剩余时间；
   - 队列前缀中必须先执行的任务工作量；
   - 对双峰/重尾桶的近似累计工作量；
   - 每个前置任务带来的 `T_host` 处理开销。

3. 对候选批估计本地批次尾延迟：

   $$T_{local}^{batch} \approx A_{oldest}^{batch} + W_{before} + \frac{\bar{E}_{batch}}{C_{src}} + T_{host}$$

   其中 `A_oldest` 是批中最老任务已经等待的时间，`\bar{E}_{batch}` 是该批任务的平均估计服务时间。

4. 计算批次风险质量：

   $$risk\_mass = \sum_{bucket} count_b \cdot \max(0, T_{local}^{b} / SLO_b - \alpha)$$

5. 若 `risk_mass <= 0` 且批次尾部估计延迟没有超过阈值，则丢弃该候选。

该步骤的关键是：估计对象是“批次风险”，不是每个任务的单独分数。

#### **0.10.8 目标主机选择与远端收益估计**

对保留下来的批候选，DQB-PM 选择远端目标：

1. 从全局 host 集合中采样 `k_dst` 个目标，排除源 host。
2. 对每个目标读取陈旧 workload：

   $$W_{dst}^{stale} = stale\_workload\_view[dst]$$

3. 加入短窗口 incoming reservation：

   $$W_{dst}^{eff} = W_{dst}^{stale} + incoming\_reservation[dst]$$

4. 结合目标容量 `C_dst` 估计远端等待：

   $$\hat{W}_{dst} \approx \frac{W_{dst}^{eff}}{cores\_per\_host}$$

5. 估计批次迁移后的远端总延迟：

   $$T_{remote}^{batch} \approx A_{oldest}^{batch} + T_{net} + T_{host} + \hat{W}_{dst} + \frac{\bar{E}_{batch}}{C_{dst}}$$

6. 计算批次收益：

   $$gain_{batch} = T_{local}^{batch} - T_{remote}^{batch} - T_{margin}$$

7. 只有 `gain_batch > 0` 且目标未超过 per-destination cap 时，目标才可用。

目标选择最终不是选“队列长度最短的 host”，而是选 stale workload、incoming reservation、容量和批次收益综合最优的 host。

#### **0.10.9 批次选择与迁移提交**

在一个 host 的 `CHECK_MIGRATION` 周期内，可能有多个 core 产生批候选。提交阶段遵循以下规则：

1. 按批次风险质量、收益、阻塞分数和源队列压力对候选做有限选择。实现上不需要全量排序，可使用固定大小 top-k、阈值选择或按 core 顺序选择。
2. 每个 host 本轮最多迁移 `batch_cap` 个任务。
3. 每个 core 本轮最多迁移 `per_core_cap` 个任务或一个批次。
4. 每个 destination 本轮最多接收 `per_dst_cap` 个任务或一批。
5. 从候选桶维护的任务指针中取出最多 `move_count` 个仍在等待队列中的任务。
6. 对每个被选任务：
   - 从源 core 等待队列中 O(1) 抽离；
   - 标记 `migrated=true`；
   - 记录 `src_host`、`dst_host`、`batch_type`、`estimated_local_latency_us`；
   - 生成远端 `TASK_ARRIVE` 事件，时间戳为 `now + T_net_oneway`。
7. 更新 `incoming_reservation[dst] += moved_batch_work_us`，直到任务到达或 reservation window 过期后释放。
8. 更新 host cooldown 和迁移预算计数。

在 MVP 中，每个迁移任务仍可对应一个 `TASK_ARRIVE` 事件；这表示 batch decision，而不是 transport-level batch。若后续实现真实 batch transport，再将同一 `(src,dst)` 的多个任务合并为一个 batch arrival。

#### **0.10.10 远端到达与执行**

迁移任务到达目标 host 后，与普通请求到达逻辑一致：

1. 目标 host 选择本地 core。
2. 若 core 空闲，任务立即执行。
3. 若 core 忙碌，任务进入等待队列，并更新目标 core 的 `QueueSummary`。
4. 对应的 `incoming_reservation` 被释放或按实际到达工作量扣减。

该步骤保证迁移任务不会绕过目标端排队规则，也不会获得不公平优先级。

#### **0.10.11 完成、指标记录与无效迁移判定**

任务完成时触发 `TASK_FINISH`：

1. 计算端到端延迟：

   $$latency = finish\_time - generate\_time$$

2. 记录 P99/P99.9 histogram、SLO violation 和任务类别。
3. 若任务曾迁移，则与迁移时估计的本地留队延迟比较：

   $$invalid = latency > estimated\_local\_latency$$

4. 更新 `migration_rate`、`invalid_migration_ratio`、按 batch type 分组的收益统计。
5. 记录 short/long 或 mice/elephant 分组 SLO violation，用于证明算法是否真的缓解了特定分布模式下的尾延迟。

#### **0.10.12 保护机制与降级路径**

DQB-PM 必须有明确降级逻辑：

1. **预算保护**：若全局迁移率接近上限，降低 batch cap 或跳过迁移。
2. **目标保护**：若目标 incoming reservation 高，禁止继续迁入。
3. **饱和保护**：若全局 P25/P50 workload 高，说明系统整体接近饱和，主动提高 margin 或进入 no-migrate。
4. **无效迁移保护**：若某窗口 invalid migration ratio 升高，说明收益估计不可靠，临时提高 margin、降低 `k_dst` 或降低 batch cap。
5. **估计误差保护**：当服务时间估计处于 `Noisy-E` 模式且误差较大时，算法应偏保守，只迁移高风险、高收益批次。

这些保护机制决定了 DQB-PM 的论文定位：它不是追求每个任务最优，而是在微秒级控制预算内做保守、近似、可解释的队列修复。

### **0.11 专家决策：关键细节处理原则**

本节用于处理当前算法最容易被质疑的细节。后续实现和论文写作应以这些决策为准。

#### **0.11.1 最小控制单位：分布区域 / 微批次**

DQB-PM 的最小控制决策单位不是单个 task，而是 **distribution region** 或 **micro-batch**。单个 task 仍然是执行、完成和统计的基本实体，但控制面不能逐 task 判断是否迁移。

一个 micro-batch 由以下维度定义：

```text
micro_batch = (source_host, source_core, service_class, arrival_epoch, risk_region)
```

- `service_class` 表示服务时间类别，例如 short、medium、long、elephant。
- `arrival_epoch` 表示请求进入该核心队列的微时间窗口，例如 2us 或 5us。
- `risk_region` 表示队列位置或风险区域，例如被长任务阻塞区、重尾阻塞区、慢节点压力区。

这样定义后，batch 的生命周期是在线可维护的：

1. 请求到达核心队列时，根据服务类别和当前 arrival epoch 加入一个 micro-batch。
2. arrival epoch 结束后，该 micro-batch 关闭，不再加入新请求。
3. micro-batch 随队列推进而被执行、部分迁移或过期删除。
4. 若 batch 太大，只能按 FIFO 前缀或工作量上限拆成 sub-batch，不允许逐 task 打分拆分。

#### **0.11.2 风险批次与迁移批次分离**

算法需要区分两个概念：

- **风险批次（Risk Batch）**：用于判断这组请求如果留在本地是否会越过 SLO 风险阈值。
- **迁移批次（Move Batch）**：实际从风险批次中迁走的 FIFO 前缀或工作量前缀。

例如，一个 short risk batch 可能包含 50 个请求，但目标端只允许安全接收 8 个请求或 40us 工作量，那么 move batch 只能是这 50 个请求中最早到达的一段前缀。算法不对 50 个请求逐一算分，也不从中挑选“分数最高”的请求。

#### **0.11.3 Batch Descriptor 的必要字段**

每个 micro-batch 至少维护以下聚合字段：

```text
count
total_estimated_work_us
mean_estimated_service_us
oldest_age_us
min_slack_us
front_blocking_work_us
service_class
risk_region_type
estimate_confidence
fifo_head_pointer / fifo_tail_pointer
```

其中 `fifo_head_pointer` 和 `fifo_tail_pointer` 只用于执行迁移时快速取出 FIFO 前缀，不用于逐 task 排序。`estimate_confidence` 用于处理服务时间估计误差，置信度越低，迁移门槛越高。

#### **0.11.4 不同分布下的统一处理方式**

实现上不应维护完全不同的算法分支，而应采用统一的服务时间分桶，再由不同模式解释这些桶：

- 双峰负载下，`short` 和 `long` 桶最重要，重点识别 `ShortBehindLong`。
- 重尾负载下，`mice` 和 `elephant` 桶最重要，重点识别 `MiceBehindElephant`。
- 异构负载下，按核心容量归一化后的 batch work 最重要，重点识别 `SlowNodeBatchPressure`。

因此，系统不一定需要先精确判断“当前负载是双峰还是重尾”。它只需要维护统一的服务时间桶，并从桶的形态中识别风险。

#### **0.11.5 远端主机选择必须以 batch-level SLO slack 为中心**

远端目标选择不能只看队列长度，也不能只看平均 workload。目标必须满足：

```text
estimated_remote_finish_time < batch_deadline
estimated_gain > migration_margin
incoming_reservation(dst) < reservation_limit
target_harm < harm_threshold
```

其中：

- `batch_deadline` 由该 batch 的 SLO 类别和最老请求 age 推导。
- `estimated_remote_finish_time` 包含网络迁移开销、目标接收开销、目标端有效等待和 batch 自身执行时间。
- `target_harm` 表示迁移该 batch 对目标端已有短任务或高优先级任务的潜在伤害。

如果没有任何目标满足这些条件，则该批次不迁移。DQB-PM 的原则是“远端有足够把握救回来才迁移”，不是“本地危险就迁移”。

#### **0.11.6 避免羊群效应的目标端保护**

羊群效应主要来自陈旧视图：多个源主机同时看到同一个目标很空，并同时迁入任务。DQB-PM 必须使用多层目标端保护：

1. **Incoming reservation**：迁移一旦被提交，立刻把该批次工作量加入目标的在途预留。
2. **Per-destination cap**：每个目标在短窗口内最多接收固定数量请求或固定工作量。
3. **Randomized target sampling**：每个源只随机采样少量目标，在候选内选择最优，避免所有源追逐同一个全局最小目标。
4. **Target hotness penalty**：若某个目标近期频繁被选中，即使陈旧 workload 仍低，也应增加惩罚。
5. **Commit-time reservation update**：一个源主机本轮连续提交多个 batch 时，每提交一个 batch 都必须立即更新 reservation，再评估下一个 batch。

MVP 阶段至少实现 host-level incoming reservation；增强版再考虑 core-level incoming reservation。

#### **0.11.7 可迁移性与顺序语义**

DQB-PM 只适用于可转发的等待请求。默认适用范围为：

- stateless 或弱状态 RPC；
- replicated service；
- 请求尚未开始执行；
- payload 或 request descriptor 较小；
- 迁移不会破坏应用语义。

对于强状态绑定、必须访问本地 shard、同一 flow/key 需要严格顺序的请求，应标记为不可迁移。实验中应加入 `migration_eligibility` 参数，例如 100%、75%、50%，验证并非所有请求都可迁移时算法是否仍然有效。

若存在顺序语义，则同一 flow/key 的请求必须作为不可拆 batch，或者禁用迁移。不能把同一顺序流中的部分请求迁走而破坏完成顺序。

#### **0.11.8 服务时间估计误差与保守策略**

DQB-PM 不能依赖真实服务时间。算法可见的是估计服务时间，来源可以是 RPC 类型、历史 EWMA、payload 大小、handler class 或应用提供的 request class。实验必须至少包含：

- `Oracle-E`：真实服务时间，仅作为上界和消融。
- `EWMA-E`：按请求类别维护历史估计，作为主实验口径。
- `Noisy-E`：加入估计误差，测试鲁棒性。

估计误差会放大 batch 迁移风险，因此应使用保守策略：

```text
estimate_confidence 越低 -> margin 越高
estimate_confidence 越低 -> batch_cap 越低
estimate_confidence 越低 -> 只迁移 risk_mass 明显更高的 batch
```

#### **0.11.9 迁移预算必须同时按请求数和工作量统计**

只报告 `migrated_requests / total_requests` 不够，因为迁移一个 elephant 任务和迁移多个 short 任务的系统影响不同。必须同时报告：

```text
migration_count_rate = migrated_request_count / total_request_count
migration_work_rate  = migrated_estimated_work / total_estimated_work
```

若未来模型加入 payload size，还应报告 `migration_byte_rate`。主结论不能只依赖低 request-count migration rate。

#### **0.11.10 控制面开销必须进入模型**

控制面不能免费。至少需要建模以下开销：

- 更新 `QueueSummary` 的开销；
- 检查 batch descriptor 的开销；
- 目标采样与远端收益估计开销；
- 提交迁移和更新 reservation 的开销。

仿真中可以先把这些开销折算为 `control_decision_cost_est_us` 和 `summary_update_cost_est_us`，再用 CloudLab c6525-25g microbenchmark 校准。若这些开销显著高于初始假设，必须使用校准值重跑主结果。

#### **0.11.11 在线可见信息边界**

算法只能使用在线可得信息：

```text
允许使用：
  本地实时队列摘要
  本地 running task 剩余时间
  远端陈旧 workload 视图
  incoming reservation
  历史服务时间估计

禁止使用：
  远端实时队列
  未来到达
  真实服务时间
  任务最终完成时间
  全局 oracle 最优目标
```

该边界必须在论文中明确，否则仿真结果容易被认为依赖不可实现信息。

#### **0.11.12 推荐的最终算法表述**

最终论文中建议这样表述算法：

> DQB-PM 将每个核心等待队列划分为若干由服务时间类别、到达时间窗口和风险区域定义的微批次。每个微批次维护数量、累计工作量、最老等待时间、最小 SLO slack 和估计置信度等摘要。控制面只检查这些批次摘要，识别长任务阻塞短任务、重尾任务压制小任务、慢节点批量积压等风险形态。若某一批次在本地预计越过 SLO 风险阈值，算法再根据陈旧远端 workload、目标容量、迁入预留、目标端伤害和 SLO slack 选择一个最可能救回该批次的远端主机，并以 FIFO 前缀形式迁移有界工作量。整个过程不对队列中所有任务逐一打分或排序。

### **0.12 专家决策：迁移粒度与传输模型**

针对“逐个任务迁移是否更好，还是一批一批迁移效率更高”的问题，当前推荐结论是：

> **控制面必须做批次级决策；MVP 阶段采用批次选择 + 逐请求转发；Plus 阶段再加入真实 batch transport 并用 c6525-25g 实测校准。**

这个取舍最稳妥。逐任务打分和排序在微秒级尾延迟系统中不可作为主算法，因为它的控制开销随队列长度增长，并且会把论文创新点退化成启发式挑任务。相反，DQB-PM 的核心应是只检查固定数量的队列摘要和微批次描述符，用分布状态和排队论判断“哪一段队列已经形成批次级 SLO 风险”。单个请求仍然是执行、完成和统计单位，但不是在线控制面的最小决策单位。

MVP 不建议直接声称“网络层批量打包迁移”带来收益。真实 batch transport 需要额外的 RPC 收发路径、批头解析、目标端解包、失败处理和顺序语义处理；如果没有 CloudLab c6525-25g 的 microbenchmark 支撑，审稿人会质疑收益来自未验证的传输假设。因此 MVP 中更保守的定义是：**算法一次选择一个迁移批次，但仿真事件层仍可把批内请求逐个产生远端到达事件**。这些请求共享同一个 `batch_id`、同一个目标选择结果和同一组迁移原因；这证明的是 batch-level control，而不是 transport-level batching。

#### **0.12.1 批次如何形成**

批次不应通过人为等待来“攒批”，否则会引入新的排队延迟。DQB-PM 的批次来自已经存在的核心等待队列：

1. 先按核心队列维护固定大小的分布摘要，例如短/长、mice/medium/elephant、到达时间窗口和风险区域。
2. 队列模式识别可以观察一个混合区域，例如长任务前方阻塞短任务，或者 elephant 前方压住 mice。
3. 真正迁移的 move batch 应尽量同质化，例如“被长任务阻塞的一组短任务”或“慢节点上已经接近 SLO 的 mice 批次”。
4. 批内迁移只取 FIFO 前缀或有界工作量前缀，不在批内重新逐任务打分。

因此，一个批次里不一定要包含所有类型任务。更准确的说法是：**观察窗口可以是混合的，风险判断利用整体分布形态；实际迁移批次应是由同一风险机制解释的一段请求集合**。这样既能利用分布状态，又不会退回到逐任务挑选。

#### **0.12.2 确定迁移批次的算法原则**

DQB-PM 确定迁移批次时不使用全队列排序，而使用固定数量候选和阈值选择：

1. 对每个核心读取当前队列摘要，识别是否存在 `ShortBehindLong`、`MiceBehindElephant`、`SlowNodeBatchPressure` 或 `GenericPressure`。
2. 对每种风险模式构造少量候选批次。候选批次由服务类别、到达窗口和风险区域定义，而不是由单个任务分数定义。
3. 使用排队论估计候选批次留在本地的尾部完成时间。估计量只依赖运行中任务剩余时间、队列前缀累计工作量、批次累计工作量、核心容量和最老等待时间。
4. 若批次尾部预计超过 SLO 风险阈值，且批次风险质量超过最小门限，则进入目标选择。
5. 若候选数量超过上限，只保留固定大小 top-k 或按阈值截断，避免控制面随队列长度线性膨胀。

这里的“排队论”不是为了追求精确闭式最优，而是为了给批次风险提供低开销、可解释、可复现实验的估计框架。论文中应强调 bounded-overhead approximate queue repair，而不是全局最优任务调度。

#### **0.12.3 迁移提交与远端到达**

一旦某个 move batch 被选中，目标主机只选择一次。目标选择基于 batch-level SLO slack，而不是批内每个请求单独重新选择：

1. 用远端陈旧 workload、目标容量、incoming reservation 和目标端伤害估计批次远端完成时间。
2. 只有当远端预计完成时间早于批次 deadline、收益超过 margin、目标端伤害低于阈值、reservation 未满时才提交迁移。
3. 提交后立即把该批次的累计工作量和请求数加入目标端 incoming reservation，防止多个源同时冲向同一目标。
4. 若目标端只能安全接收部分工作量，则缩小 FIFO 前缀；如果缩小后低于最小批大小，则本轮不迁移。
5. MVP 仿真中，批内每个请求可以继续对应一个远端到达事件，但这些事件由同一次批次决策产生，不再触发批内逐任务目标选择。

这条规则回答了“是不是还要继续选择批次内单个任务”的问题：**不选择单个任务，只选择批次前缀的边界**。边界由工作量上限、请求数上限、SLO slack 和目标可接收容量共同决定。

#### **0.12.4 Plus 版本的真实批量传输**

当 MVP 结果稳定后，可以加入 Plus 版本验证真实 batch transport 是否进一步降低开销。此时可以把同一 `(src_host, dst_host)` 的多个请求描述符合并成一个 `MigrationBatch`，并使用如下成本模型：

$$
T_{batch}(k)=T_{net\_base}+T_{batch\_header}+k \cdot T_{request\_descriptor}+T_{batch\_unpack}
$$

其中 $k$ 是批内请求数量。这个模型必须通过 c6525-25g microbenchmark 校准，包括单请求转发成本、批头固定成本、每个请求描述符增量成本和目标端解包成本。

Plus 版本只在实测表明确有固定成本可摊销时作为增强贡献。如果 batch transport 的收益很小，论文仍然可以成立，因为主贡献是分布感知的批次级控制和队列修复，而不是传输层打包优化。

## **一、 仿真器全局架构设计**

为了实现高吞吐、低开销且时序精准的离散事件仿真，底层架构必须高度优化内存管理和事件调度效率。本仿真器采用基于进程/资源视角（Process/Resource View）的纯事件驱动机制，摒弃固定时间步长（Fixed Time-step）模型，以支持千万级/秒的高密度微秒级事件吞吐 2。

### **1.1 核心类结构与抽象定义**

仿真器的核心骨架建立在高度解耦的基础类群之上。这些类共同构成了数据中心计算与网络资源拓扑的数字孪生。

| 核心类名 | 设计规范与内部状态 | 物理映射与指代功能 |
| :---- | :---- | :---- |
| Simulator | 全局单例（Singleton）。内部包含一个基于 std::priority\_queue 的最小堆 event\_queue，以及全局虚拟时钟 current\_time。需实现 schedule\_event() 和 run()。 | 物理宇宙的时间流逝与事件仲裁者。维护全局网络拓扑延迟矩阵与系统宏观状态（如已完成任务总数）。 |
| Node | 包含 std::vector\<Core\*\>、节点级网卡（NIC）接收队列。维护节点级状态，如 node\_id、总并发连接数。 | 数据中心内的一台独立物理服务器（Host）。负责网卡层面的流量接收与第一层分发（如 RSS 机制模拟）。 |
| Core | 包含就绪任务队列（需支持 O(1) 中部抽离，后文详述）。维护 is\_idle 状态标志、current\_task 指针、以及代表异构算力的系数 $C\_j$（标准节点 $C\_j=1.0$，慢节点 $C\_j=0.2$）。 | 独立的逻辑 CPU 核心。实际执行请求的物理资源单元，直接影响任务的服务时间（Service Time）。 |
| Task | 包含元数据：task\_id，generate\_time，arrive\_time，base\_service\_time，slo\_threshold。包含一维时间戳数组用于全链路追踪。 | 微秒级 RPC 请求。在仿真中不携带真实 Payload，仅作为排队论与调度算法计算的统计实体。 |
| Event | 抽象基类。包含 timestamp、EventType 和 priority。核心纯虚函数为 virtual void process() \= 0;。派生类将实现具体的业务逻辑。 | 离散时间点上发生的状态突变。驱动 Simulator 推进虚拟时间的唯一动力 4。 |

### **1.2 高性能最小堆队列与内存池设计**

由于微秒级负载压测将在短时间内产生数以亿计的事件，直接使用原生的 new 和 delete 会引发严重的内存碎片（Memory Fragmentation）以及系统调用阻塞，从而导致仿真器运行极度缓慢。

**内存对象池（Memory Pool）机制**： 对于 Task 和各类 Event，必须在 Simulator 启动时预分配连续的大块内存（如使用 std::pmr::monotonic\_buffer\_resource 或自定义的 Free-List 结构）。在事件触发与销毁时，通过重载 operator new 和 operator delete 将内存归还至池中 6。这不仅将内存分配的时间复杂度降为 $O(1)$，同时极大提升了 CPU L1/L2 缓存的命中率（Cache Locality）。

**全局事件队列管理**：

优先队列的比较器 EventComparator 必须经过精心设计，不仅要保证时间戳越小的事件越早出队（最小堆），还要处理时间戳完全相同时的并发决断逻辑。

为了确保物理逻辑的因果一致性，同等时间戳下的事件优先级应为：TASK\_FINISH \> TASK\_ARRIVE \> TASK\_GENERATE。这保证了资源的释放总是优先于新请求的竞争。

## **二、 离散事件生命周期（状态机）**

微秒级调度的保真度，不仅取决于对处理时间的模拟，更在于对网络协议栈和调度开销（Overhead）的精确建模 1。仿真器内定义了以下六大核心事件，其状态机流转必须严丝合缝地对齐物理机实测的延迟约束。

### **2.1 核心事件的触发条件与执行逻辑**

**1\. TASK\_GENERATE (负载生成事件)**

* **触发条件**：按预设的到达过程（如泊松过程 Poisson Process，到达间隔 $\\Delta t \\sim Exp(\\lambda)$）递归生成下一个到达事件 7。  
* **物理约束处理**：支持生成混合长尾分布（如 Bimodal）的基础服务时间 E。如果生成的请求来源于客户端且需经过网络，则当前不产生计算负载，而是立刻向系统调度一个目标为目标节点的 TASK\_ARRIVE 事件。该到达事件的时间戳必须加上单向网络延迟 $3.15 \\mu s$。

**2\. TASK\_ARRIVE (网络到达核心队列事件)**

* **触发条件**：请求通过网卡进入目标节点的操作系统边界，由 NIC 经过网卡多队列（如 RSS 哈希）分发或由节点全局分配器分配。  
* **逻辑流转**：  
  * 读取目标 Core 的状态。  
  * 若 Core-\>is\_idle \== true，直接将任务送入执行态，生成一个时间戳为 current\_time 的 TASK\_EXECUTE 事件。  
  * 若 Core 处于忙碌状态，则将任务插入该 Core 的等待队列尾部。此时必须更新该队列的累计总工作量（Total Pending Work），以供主动迁移算法探测。

**3\. TASK\_EXECUTE (任务开始执行事件)**

* **触发条件**：TASK\_ARRIVE 发现空闲核心，或上一个 TASK\_FINISH 事件主动拉取队列中的下一个任务。  
* **物理约束处理**：核心开始执行时，必须计算真实的物理耗时 $T\_{exec}$。根据物理机实测，考虑到 DPDK 轮询开销与上下文切换，前置处理时间加上实际执行时间为：$T\_{exec} \= \\frac{E}{C\_j} \+ T\_{host}$。其中 $C\_j$ 为当前核心的异构算力系数，$T\_{host} \= 2.1 \\mu s$。  
* **逻辑流转**：将 Core 设为 busy，记录当前正在执行的 current\_task，并向未来时间轴调度一个时间戳为 current\_time \+ T\_{exec} 的 TASK\_FINISH 事件。

**4\. TASK\_FINISH (执行完毕事件)**

* **逻辑流转**：  
  * 任务生命周期结束。计算端到端延迟（$T\_{finish} \- T\_{generate}$），将数据送入底层 Metrics Collector。  
  * 将 Core 状态置为 idle，清空 current\_task。  
  * **触发后续**：若该核心所在的本地队列非空，则立刻弹出队首任务，生成时间戳为 current\_time 的新 TASK\_EXECUTE 事件。  
  * **Baseline 特化逻辑**：如果在 ZygOS 模式下，本地队列为空，则启动工作窃取（Work-stealing）状态机，向其他队列广播窃取探针。

**5\. SYNC\_LOAD (状态同步心跳事件)**

* **物理背景**：分布式集群中不存在上帝视角（Global Omniscience），节点间对彼此负载的认知存在固有的网络传播延迟。  
* **逻辑流转**：系统定时器按一定频率（如每 $10\\mu s$ 或 $50\\mu s$）触发。所有节点通过模拟的控制平面更新其对其他节点队列深度的视图。由于微秒级任务生命周期极短，当一个节点在 CHECK\_MIGRATION 依赖其他节点的负载视图时，所引用的数据必然是陈旧的（Stale Information）。这种模拟确保了仿真中的主动迁移算法必须经受“陈旧状态抖动”的考验 1。

**6\. CHECK\_MIGRATION (主动迁移扫描事件 \- 算法挂载点)**

* **触发条件**：可配置为周期性扫描（如每 $1 \\mu s$ 触发一次），或在 TASK\_ARRIVE 导致本地队列长度突破危险阈值时作为中断事件触发。  
* **核心动作**：遍历本地就绪队列，执行主动预测算法（下文详解）。若决定迁移某个处于队列中部的任务，则立即从队列中将其精准抽离，并向目标更优节点调度一个 TASK\_ARRIVE 事件，时间戳设为 current\_time \+ 3.15 \\mu s。

## **三、 核心算法：主动迁移逻辑实现指导**

主动预测迁移的核心思想是“防患于未然”。现有的硬件调度和内核调度只能感知队列长度（Queue Length），而在高离散长尾分布（High-dispersion Workloads）下，队列长度无法真实反映任务的等待时间 1。必须引入基于微秒排队论预测的确定性数学模型。

### **3.1 本地队列预计等待时间（$W\_i$）的排队论建模**

在 M/G/1 或 M/M/c 等排队系统中，对于刚刚到达系统的新任务，其排队等待时间 $W\_q$ 通常使用 Pollaczek-Khinchine (P-K) 公式或 Erlang-C 公式进行宏观平均预期 7。然而，主动迁移需要的是对**单个特定任务 $i$** 在确切时刻的局部等待时间预估。

由于在离散事件仿真中，本地节点的微观状态是完全可观测的（Residual Work 确知），我们不再使用稳态均值近似，而是采用确定的瞬态方程。

设目标任务 **$i$** 在当前逻辑核 Core 的队列中排在第 **$k$** 位（即前面有 $k-1$ 个就绪任务）。

设当前 Core 正在执行的任务的剩余执行时间为 $R\_{running}$。

设前方排队的任务集合为 $Q\_{head} \= \\{T\_1, T\_2,..., T\_{k-1}\\}$。当前核心的算力系数为 $C\_{local}$。

若系统具有服务时间先验知识（如通过历史请求类型探测 EWMA 预估得出 $E$），则任务 $i$ 在本地的确切预计等待时间 $W\_i$ 为：

$$W\_i \= \\frac{1}{C\_{local}} \\left( R\_{running} \+ \\sum\_{m=1}^{k-1} E \\right) \+ (k-1) \\times T\_{host}$$

其中，$(k-1) \\times T\_{host}$ 极度关键。每个排在前面的任务在结束和切换时，都会导致 $2.1 \\mu s$ 的协议栈与调度底噪。如果不将这一开销纳入计算，将导致对本地等待时间的严重低估，进而错过最佳的主动迁移时机。

### **3.2 结合 $3.15 \\mu s$ 网络硬约束的迁移决策方程**

单向跨机迁移的刚性惩罚为 $3.15 \\mu s$ 的网络延迟加上到达远端后不可避免的  远端$2.1 \\mu s$主机接收底噪。总计 $5.25 \\mu s$ 的固定惩罚意味着迁移并不是没有成本的。

系统不仅要保证任务迁移后总耗时变短，还要确保迁移是**有必要的**（即如果不迁移，当前任务将违反 SLO）。

设远端候选节点 $d$ 预估等待时间为 $\\hat{W}\_d$，其算力系数为 $C\_d$。任务的 SLO 宽容阈值为 $T\_{slo}$。任务已存在于系统的时间为 $T\_{elapsed} \= current\\\_time \- T\_{generate}$。

**主动迁移的三重判定准则**：

1. **违约风险约束**：预期本地总驻留时间逼近 SLO（例如达到 80% 的风险水位）。  
   $$T\_{elapsed} \+ W\_i \+ \\frac{E}{C\_{local}} \> T\_{slo} \\times \\alpha$$

2. **绝对收益约束**：远端执行的总预期时间必须严格小于本地预期时间。  
3. **网络惩罚与防抖动约束（Hysteresis）**：必须覆盖迁移的刚性成本，并超出一定安全冗余 $T\_{margin}$，以对抗网络陈旧状态（Stale Information）导致的“乒乓迁移”。  
   $$W\_i \+ \\frac{E}{C\_{local}} \> \\hat{W}\_d \+ 3.15\\mu s \+ T\_{host} \+ \\frac{E}{C\_d} \+ T\_{margin}$$

**C++ 算法伪代码实现规范**：

C++

void Core::evaluate\_and\_execute\_proactive\_migration() {  
    double current\_time \= Simulator::get\_instance()-\>now();  
    double cumulative\_wait \= this\-\>get\_residual\_running\_time();  
      
    // 从队列头部向尾部扫描  
    auto it \= this\-\>intrusive\_task\_list.begin();  
    while (it\!= this\-\>intrusive\_task\_list.end()) {  
        Task\* task \= &(\*it);  
          
        // 累加当前任务自身的预期时间（作为后续任务的等待时间）  
        double current\_task\_cost \= (task-\>expected\_service\_time / this\-\>capacity\_factor) \+ 2.1;  
        cumulative\_wait \+= current\_task\_cost;  
          
        // 判定是否面临 SLO 违约风险  
        double local\_total\_time \= (current\_time \- task-\>generate\_time) \+ cumulative\_wait;  
        if (local\_total\_time \> task-\>slo\_threshold \* 0.8) {  
              
            // 查询全局视图获取最优远端 Node（受制于 SYNC\_LOAD 的陈旧数据）  
            Node\* best\_remote \= GlobalView::get\_best\_remote\_node(task-\>expected\_service\_time);  
            if (best\_remote\!= nullptr) {  
                double remote\_wait \= best\_remote-\>get\_estimated\_wait\_time();  
                double remote\_exec \= task-\>expected\_service\_time / best\_remote-\>capacity\_factor;  
                  
                // 3.15us 网络底噪硬约束判定  
                double remote\_total\_time \= (current\_time \- task-\>generate\_time) \+   
                                           3.15 \+ 2.1 \+ remote\_wait \+ remote\_exec;  
                                             
                if (local\_total\_time \> remote\_total\_time \+ MARGIN\_THRESHOLD) {  
                    // 执行抽离：生成远端到达事件  
                    Simulator::get\_instance()-\>schedule\_event(  
                        new Event\_TaskArrive(current\_time \+ 3.15, best\_remote, task)  
                    );  
                      
                    // O(1) 中部抽离维护逻辑：从当前累加等待时间中扣除被抽离任务的开销  
                    cumulative\_wait \-= current\_task\_cost;  
                    // 从数据结构中安全抹除并指向下一个元素  
                    it \= this\-\>intrusive\_task\_list.erase(it);   
                    continue;   
                }  
            }  
        }  
        \++it;  
    }  
}

### **3.3 分布感知队列批迁移（DQB-PM）设计修订**

上述逐任务扫描伪代码可作为 `M0` 基线保留，但不应作为最终投稿算法的核心叙事。微秒级 RPC 系统中任务数量巨大，如果每个控制周期都对等待队列中的任务逐一估计、逐一打分并排序，控制面开销会与队列长度线性相关，难以满足微秒级尾延迟约束。最终算法应改为 **Distribution-aware Queue-Batch Proactive Migration（DQB-PM）**：每个核心对本地等待队列维护固定大小的分布摘要，并以“队列批次”而不是“单个任务”为迁移决策对象。

**核心思想**：

1. **用分布桶替代逐任务打分**。对于双峰负载，队列摘要至少维护 `short_count`、`long_count`、`short_work_us`、`long_work_us`、`oldest_short_age_us`、`oldest_long_age_us`。对于重尾负载，队列摘要可维护 `mice / medium / elephant` 三类任务的数量、累计工作量与最老任务等待时间。桶数量是常数，因此控制路径复杂度与队列长度解耦。
2. **用排队论估计批次风险**。对某个核心队列，先计算正在运行任务的剩余时间 $R$，再计算各分布桶的累计服务需求。对于短任务批，可估计：

   $$T\_{batch}^{short} \approx A\_{oldest}^{short} + R + W\_{before}^{short} + \frac{\bar{E}\_{short}}{C\_{src}} + T\_{host}$$

   其中 $A\_{oldest}^{short}$ 是该批中最老短任务已经等待的时间，$W\_{before}^{short}$ 是由队列前缀和长任务阻塞造成的预计等待时间。若 $T\_{batch}^{short} > \alpha \cdot SLO\_{short}$，则生成一个短任务批迁移候选。
3. **识别分布特定的风险模式**。双峰分布下重点识别 `ShortBehindLong`：长任务处于队列前部或正在运行，后续短任务批即将因 HoL 阻塞违约。重尾分布下重点识别 `MiceBehindElephant` 或 `ElephantDominatedQueue`：少数超长任务占据大部分 workload，使小任务尾延迟恶化。
4. **以批为单位选择目标**。远端收益不再对每个任务分别计算，而是对批次的平均服务时间、总工作量、最紧 SLO slack 和目标主机陈旧 workload 进行估计。目标选择必须叠加短窗口 `incoming_reservation`，避免多个源主机同时把批次迁往同一个看似空闲的目标。
5. **只迁移等待队列中的 RPC 元数据**。本文中的迁移语义是 queued request forwarding，不迁移正在执行的线程、内存页或部分执行状态。该边界必须在论文中明确说明，否则迁移开销会被严重质疑。

**建议的批候选结构**：

```cpp
struct QueueBatchCandidate {
    int src_host = -1;
    int src_core = -1;
    int short_count = 0;
    int long_count = 0;
    int elephant_count = 0;
    int move_count = 0;
    double batch_work_us = 0.0;
    double oldest_age_us = 0.0;
    double estimated_local_tail_us = 0.0;
    double risk_mass = 0.0;
    double blocking_score = 0.0;
    BatchType type = BatchType::GenericPressure;
};
```

**客观评估**：

该方向比逐任务打分更适合 INFOCOM 论文叙事，因为它把创新点从“启发式迁移某个任务”提升为“在微秒级控制预算下，利用服务时间分布和排队论做近似队列修复”。它的优势是复杂度可控、与双峰/重尾负载机制强相关、能自然解释 HoL 阻塞；风险是批次估计可能比单任务估计更粗，若服务时间估计误差大或目标端 incoming reservation 不准确，可能把一批任务迁往错误目标。因此必须配套三类实验：`Oracle-E / EWMA-E / Noisy-E` 服务时间估计敏感性、`incoming_reservation` 消融、以及批大小和控制周期敏感性。

### **3.4 “O(1) 队列中部精准抽离”的高性能数据结构设计**

在主动迁移逻辑中，我们极有可能发现排在队列**中部**的长任务由于前方积压，面临违约风险，需要被抽离。

在标准 C++ 模板库（STL）中：

* std::queue 和 std::priority\_queue 不支持遍历和中部删除操作。  
* std::vector 或 std::deque 虽然支持随机访问，但中间删除操作（erase）会导致后续元素的内存块平移拷贝，其时间复杂度为 O(N)。在千万级 RPS 的微秒模拟中，这会导致 CPU 缓存失效（Cache Invalidations）和灾难性的性能骤降 10。  
* std::list 支持 O(1) 删除，但需要额外的 std::unordered\_map 来跟踪节点指针，维护哈希表的开销在极高频访问下同样不可接受 12。

**工程最优解：侵入式双向链表（Intrusive Doubly-Linked List）** 为满足极端仿真性能，必须采用侵入式容器设计（如 boost::intrusive::list，或手写侵入式链表）13。 设计机制如下：

1. 直接在 Task 类的内部声明 next 和 prev 指针。  
   C++  
   struct Task {  
       Task\* next \= nullptr;  
       Task\* prev \= nullptr;  
       //... 其他属性  
   };

2. 调度器持有一个链表头和尾指针构成的 IntrusiveList 类。  
3. 当迭代器定位到中部面临风险的 Task\* 时，直接通过解链操作（task-\>prev-\>next \= task-\>next 等）完成从队列的摘除，纯粹的指针解引用操作耗时极低，绝对保证 O(1) 时间复杂度且无需任何动态内存分配（由于此时任务对象 Task 的物理内存并不销毁，而是被包裹进一个新的 TASK\_ARRIVE 事件发往远端）。这种设计彻底消除了大规模内存操作对 DES 性能的影响。

## **四、 Baseline（基准测试）理论模型与工程对齐指导**

顶级研究中证明优越性的前提是：基准对比必须绝对公平，且不能在代码实现中刻意劣化 Baseline。仿真器需通过预配置枚举 Config::SCHEDULER\_TYPE 自由切换，重现领域内最经典的三大范式。

### **4.1 Ideal cFCFS (理想全局单队列)**

对标文献中（如 Shinjuku 1、Draconis 1）针对极其轻量级任务所推崇的理想单队列。

* **架构形态**：消除单个节点或核心的本地队列。整个集群唯独维护一个全局大队列 Simulator::global\_queue。  
* **代码调度逻辑**：采用经典的拉取（Pull-based）模型。一旦任何节点的任何核心抛出 TASK\_FINISH 事件变为空闲，立刻向全局调度器发送拉取请求。  
* **硬约束模拟惩罚**：在此模式下，绝无因局部分布不均带来的 Head-of-Line (HoL) 阻塞。然而，为确保物理公平，**每一次全局调度拉取必须被施加物理通信惩罚**。拉取任务等同于一次额外的网络往返，当全局队列分配任务给某节点时，在 current\_time 的基础上强制叠加 $3.15 \\mu s$ 的网络延迟与 $2.1 \\mu s$ 主机接收开销，使得任务达到真正的可用状态。  
* **验证现象**：在极端短任务（例如平均服务时间 $2 \\mu s$）压测下，cFCFS 的通信开销将大幅超越任务本身的执行时间，导致吞吐量崩溃，从而侧面烘托主动迁移（Local Queue为主，辅以例外干预）在减少通信开销上的绝对优势。

### **4.2 Power-of-2-Choices (仅在分发层随机做决定)**

分布式调度与机架级硬件卸载（如 RackSched 1 和 R2P2 1 变体）最常用的算法，旨在解决全局单队列的瓶颈。

* **架构形态**：节点间解耦，每个核心或节点独立维护等待队列。  
* **代码调度逻辑**：  
  * 在触发 TASK\_GENERATE 时，分发器利用 std::mt19937\_64 伪随机抽取两个远端节点（Node A 和 Node B）。  
  * 读取这两个节点的队列长度状态。**必须注意**：此时读取的并非节点真正的实时长度，而是全局维护的、由 SYNC\_LOAD 事件周期性更新的**陈旧状态矩阵**（Stale Load State）1。  
  * 根据陈旧状态判断较短的队列，发起 TASK\_ARRIVE 并施加 $3.15 \\mu s$ 网络分发延迟。任务落地后无法再被二次迁移。  
* **验证现象**：在同构集群中，此基准极其优秀。但一旦加入异构降频节点（慢节点），慢节点队列由于陈旧信息的误导，将遭受微爆流的轰炸（Herd Effect，羊群效应），且因无法主动纠正，导致尾延迟不可挽回地飙升。

### **4.3 ZygOS (被动 Work-stealing)**

对标利用 CPU 周期的去中心化负载均衡经典系统 1。

* **架构形态**：初始流量由硬件网卡（如基于连接的 RSS）均匀盲分发至所有核心队列。允许核心进行跨核与跨机的工作窃取。  
* **代码调度逻辑**：  
  * 在 TASK\_FINISH 事件中，若当前核心发现自身队列为空，则进入窃取循环。  
  * **本地窃取**：随机轮询同一节点内的其他核心队列。若成功，附加上下文切换和锁竞争模拟开销（$0.4 \\mu s \- 1.0 \\mu s$）。  
  * **跨机窃取**：若本地无任务可窃取，发送跨机窃取请求。这一过程不仅需要经历一来一回的 RTT（$3.15 \\mu s \\times 2 \= 6.3 \\mu s$）的时间惩罚，同时还需要对方 CPU 打断响应。  
* **验证现象**：被动窃取的本质是“事后诸葛亮”。在异构场景或突发流量中，排在拥塞队列后方的任务在漫长的等待后才会由一个空闲核心发起拯救，往往耗时已经打破了微秒级 SLO。这直接凸显出我们提出“预测违规、主动推离”机制在微秒时间尺度上的优越性。

## **五、 核心评测场景与指标记录**

系统顶会（INFOCOM）极其看重仿真对极端边角案例（Corner Cases）的覆盖和底层统计的严密性。评测框架必须全方位碾压 Baseline。

### **5.1 数据收集器设计（Metrics Collector）**

在每秒发生数百万次请求的场景下，保存每个任务的完整耗时到 std::vector\<double\> 中进行后处理排序是极不专业的，不仅消耗 GB 级的内存，甚至引发 OOM（Out of Memory）。

* **HDR Histogram 实现**：  
  仿真器必须设计并引入高动态范围直方图（High Dynamic Range Histogram）。针对微秒级任务，设置记录范围为 $1 \\mu s$ 至 $10,000 \\mu s$，有效数字精度（Significant Figures）设为 3，通过对数分桶算法，以仅仅几十 KB 的固定内存开销实现对所有落点的高精度覆盖。  
* **输出关键指标**：  
  1. **Tail Latency**：$99^{th}$ 和 $99.9^{th}$百分位尾延迟分布随系统负载的演变曲线。  
  2. **Migration Rate**：记录 $(主动迁移任务数 / 总生成任务数)$。用来反驳“主动迁移会带来网络风暴并挤占正常通信带宽”的质疑，证明该算法只需极小的迁移比例即可抚平尾延迟长尾。  
  3. **Invalid Migration Ratio**：防抖动阈值 $T\_{margin}$ 有效性探针。记录被迁移后、实际总延迟反而高于不迁移预期延迟的失误比例，验证模型精确度。

### **5.2 核心测评用例的参数配置阵列**

要在论文的 Evaluation 章节中形成完美的逻辑闭环，须在仿真器中实现以下具有递进式深度的压测场景（基于$T\_{host} \= 2.1 \\mu s$, $T\_{net} \= 3.15 \\mu s$ 硬约束）：

#### **实验一：同构极限压测（Homogeneous Stress Test）**

验证算法的基础有效性和对系统容量的榨取能力。

* **集群拓扑**：64 个物理 Node，每节点 16 个物理核 Core，集群共计 1024 核心。所有核的算力系数 $C\_j \= 1.0$。  
* **流量分布**：泊松到达过程。执行时间采用长尾混合双峰分布（Bimodal Workload 1）。配置 80% 的短任务期望执行时间 $E \= 5 \\mu s$；20% 的长任务执行时间 $E \= 100 \\mu s$。这种混合工作负载极易在盲调度的队列中制造 Head-of-line 阻塞。  
* **验证方法**：动态调整泊松到达率 $\\lambda$，驱动系统宏观负载率 $\\rho \= \\lambda \\times E / (1024 \\times C\_j)$ 从 10% 线性拉升至 95%。观测 $99^{th}$ 延迟突变拐点（Knee Point）。  
* **理论预期**：由于 ZygOS 和 Power-of-2 在高负载下失去窃取目标或产生测不准现象，尾延迟将在 80% 负载后呈指数级爆炸。而主动式迁移在预判违规时通过中段抽离直接破除 HoL 阻塞，将安全负载极限延展至 90% 以上。

#### **实验二：异构降频 \+ 瞬时突发叠加（Heterogeneous Stragglers \+ Micro-bursts）**

模拟当代真实公有云环境下的“吵闹邻居（Noisy Neighbors）”现象 1。

* **异构性植入**：动态选取 15% 的 Node，将其所有核心的$C\_j$突降为 0.2（处理耗时放大 5 倍）。  
* **突发流量建模**：禁用平滑的泊松流，改用马尔可夫调制泊松过程（Markov Modulated Poisson Process, MMPP）。状态机拥有“Normal”和“Burst”两种状态，在微秒级时间内跳转。在 Burst 状态下，局部集群或指定节点的任务到达率会瞬间冲高至额定容量的 1.5 倍，持续时间设置为 $500 \\mu s$。  
* **验证方法**：固定整体宏观负载在稳态（如 60%），但周期性引爆 Straggler 节点上的 Micro-burst。  
* **核心论点**：在这一地狱级工况下，基于被动窃取的机制在 $500 \\mu s$ 的突发期内根本反应不过来，积压在降频队列尾部的长尾将彻底失控；近似最优队列（JSQ）更会将流量源源不断推给错误节点。唯有通过本仿真项目实现的主动预测模型，不仅能提前预见灾难，更能将任务在中段强制解链（借助 O(1) 数据结构）并承受极小（$3.15 \\mu s$）代价逃向安全节点，达成降维打击的延迟曲线。

### **总结**

本文档提供了面向顶会 INFOCOM 的离散事件微秒级仿真架构的底层解构准则。从避开堆内存碎片化的侵入式内存池与 $O(1)$ 抽取结构，到融合物理机网络底噪的 $5.25 \\mu s$ 迁移硬性惩罚约束；从使用非稳态确定方程预估特定队列的排队延迟，到防抖动的预测迁移伪代码实现。严格依据本文档执行 C++ 仿真器的代码落地，不仅能够在逻辑上自洽，所得的仿真图表更具备无可辩驳的工程严密性和物理置信度。
