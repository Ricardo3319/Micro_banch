# Step-12 主机内 Core 调度实验闭环总结


## 实验目标

本阶段将研究范围从“跨主机主动迁移”收缩到“单台主机内部多个 core 之间的任务调度与完成”。核心问题是：在讨论跨主机迁移之前，是否应该先通过主机内调度机制修复 core 级队列不均衡。

本阶段暂不考虑跨主机迁移、网络转发、远端 stale view 和目标主机 reservation，只研究任务到达单台 host 后，在 16 个 core 之间如何排队、执行、工作窃取和主动迁移。

## 实验设置

实验输出文件：

- `artifacts/step-12-intra-host/intra_w3_only.csv`

实验场景：

- 工作负载：`W3_POISSON_LOGNORMAL`
- 负载强度：`rho = 0.85`
- 主机数量：`1`
- core 数量：`16`
- warmup 请求数：`200000`
- measurement 请求数：`1000000`
- 随机种子：`{11, 23, 37, 47, 59}`

对比方法：

- `L0_RandomCore`：任务到达后随机分配到某个 core，不做后续本地修复。
- `L1_WorkStealing`：当 core 空闲时，从本 host 内剩余工作量最大的 core 窃取一个等待任务。
- `M0_IntraHostProactive`：周期性扫描本 host 内等待队列，对预计存在 SLO 风险且迁移后有收益的等待任务执行主动迁移。

已执行验证命令：

```powershell
cmake --build build-aqb-check
.\build-aqb-check\simulator.exe intra-smoke
.\build-aqb-check\simulator.exe regression
.\build-aqb-check\simulator.exe intra-w3-only
```

说明：`intra-main` 完整 rho sweep 曾尝试运行，但超过本地 5 分钟命令超时时间，未将其不完整结果纳入本次分析。

## 实验结果

以下结果为 5 个随机种子的中位数：

| 方法 | P99/us | P999/us | SLO 违约率 | 主机内移动率 | 移动任务数 | 移动工作量/us | 无效主机内迁移比例 |
|---|---:|---:|---:|---:|---:|---:|---:|
| `L0_RandomCore` | 2050 | 3260 | 0.761112 | 0 | 0 | 0 | 0 |
| `L1_WorkStealing` | 220 | 364 | 0.203070 | 0.328258 | 393913 | 12520900 | 0 |
| `M0_IntraHostProactive` | 194 | 348 | 0.153335 | 0.399925 | 479917 | 10274200 | 0 |

相对改进幅度：

| 对比 | P99 降低 | P999 降低 | SLO 违约率降低 |
|---|---:|---:|---:|
| `L1_WorkStealing` 相比 `L0_RandomCore` | 89.27% | 88.83% | 73.32% |
| `M0_IntraHostProactive` 相比 `L1_WorkStealing` | 11.82% | 4.40% | 24.49% |
| `M0_IntraHostProactive` 相比 `L0_RandomCore` | 90.54% | 89.33% | 79.85% |

## 结果分析

第一，主机内 core 调度是当前问题中的基础环节，不能跳过。随机 core 分配在 W3 重尾服务时间下会造成严重的 core 级队列不均衡，中位数 P99 达到 `2050us`，P999 达到 `3260us`。这说明尾延迟的主要来源之一不是跨主机负载不均衡，而是单台主机内部 core 队列被长任务阻塞后形成的局部积压。

第二，工作窃取是必须纳入的强基线。`L1_WorkStealing` 将 P99 从 `2050us` 降低到 `220us`，将 P999 从 `3260us` 降低到 `364us`。这证明在主机内存在空闲 core 时，只要允许空闲 core 从高负载 core 窃取等待任务，就能显著缓解重尾任务造成的局部阻塞。

第三，主动迁移在工作窃取之上仍然有增益。`M0_IntraHostProactive` 将 P99 从 `220us` 进一步降低到 `194us`，将 P999 从 `364us` 降低到 `348us`，同时将 SLO 违约率从 `0.203070` 降低到 `0.153335`。这说明主动迁移的价值在于：它不必等到某个 core 空闲后才触发，而是可以提前发现存在 SLO 风险的等待任务，并迁移到当前预计完成时间更低的 core。

第四，主动迁移移动的任务数更多，但移动的总工作量反而低于工作窃取。工作窃取的中位移动工作量为 `12520900us`，主动迁移为 `10274200us`。这表明主动迁移更倾向于移动被长任务阻塞的短任务或中等任务，而不是简单搬运最多工作量。这一点符合“风险预测迁移”的设计目标。

第五，本阶段所有方法的跨主机 `migration_rate` 均为 `0`，主动迁移的 `invalid_intra_move_ratio` 也为 `0`。因此，本阶段结果可以明确解释为单 host 内部 core 调度实验结果，不混入跨主机迁移因素。

## 闭环结论

本阶段实验形成如下闭环结论：

```text
在 W3 重尾任务场景下，尾延迟首先表现为单台主机内部 core 队列不均衡问题。
工作窃取能够大幅降低 P99/P999，是后续算法必须对比的基础方法。
基于风险预测的主机内主动迁移在工作窃取基础上仍能进一步降低尾延迟和 SLO 违约率，
说明主动迁移的合理应用场景应先放在主机内 core 间任务调度，而不是直接跨主机迁移。
```

因此，当前课题后续应优先围绕主机内 core 调度展开：

- 将工作窃取作为核心对比算法，而不是只与跨主机迁移类方法比较。
- 将主动迁移算法定位为主机内风险任务提前修复机制。
- 跨主机迁移暂时作为后续扩展，不作为当前阶段主实验。

## 后续工作

当前闭环只完成了 W3、`rho=0.85` 的重点实验。下一步应补充更完整的负载扫描和稳定性分析。

建议下一阶段实验矩阵：

- W3 重尾任务：`rho = 0.50, 0.70, 0.85, 0.92`
- W1 双峰任务：`rho = 0.85`，用于验证简单负载下是否退化
- 高负载压力点：`rho = 0.95`，用于验证主动迁移是否会过度搬运

下一阶段验收标准：

- 工作窃取必须作为主机内调度基线保留。
- 主动迁移在 W3 主实验点上应至少在 P99、P999 或 SLO 违约率之一超过工作窃取。
- 所有主机内实验模式必须保持跨主机迁移率为 `0`。
