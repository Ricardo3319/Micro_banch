# Step-01 Tier-1 迭代总结与根因分析

> 环境：cmake 4.2.3 + g++ 15.2.0 (MSYS2 ucrt64) + ninja 1.13.2，Windows 11  
> 集群：64 hosts × 16 cores = 1024 cores，均质 C=1.0  
> 工作负载 W2：MMPP(2-state) + Bimodal（80%×5us + 20%×100us），E[S]=24us  
> 统计：warmup=200k，measurement=1M，seeds={11,23,37,47,59}  
> 验收标准：≥2 个 rho 点使 M0 相对 B2 的 P99 改善 >5%，且 imr≤0.30，mr≤0.05

## 一、全部迭代记录

### 迭代 1–2：公平性修复 + 迁移风暴治理

| 修改项 | 内容 |
|--------|------|
| 问题 | M0 直接读取远端主机的 **实时** 队列长度，违反公平性约束 |

## 三、根因分析：M0 为什么无法在高负载下胜过 B2？

### 3.1 架构级不对称：B2 用实时本地信息决策 vs M0 依赖陈旧远端信息

| 维度 | B2 (Reactive) | M0 (Proactive) |
|------|---------------|----------------|
| 迁移触发条件 | `local_total_queue_len() >= Q_hi` — **实时本地数据** | `local_total_us > SLO * alpha` — **实时本地数据** ✓ |
| 目标节点选择 | Power-of-K stale_view，需 `≤ Q_lo` | Power-of-K stale_view |
| 迁移收益判定 | **无**（仅阈值判断，不估算远端等待） | `remote_total < local_total + margin` — **依赖 stale_view 估算远端** ✗ |
| 关键弱点 | — | **远端等待时间基于 10us 前的陈旧快照** |

B2 的决策仅依赖本地实时状态（`Q > Q_hi` 且存在 `Q_lo` 目标），**不尝试精确估算远端等待时间**。它的"迁移收益"判断是隐式的：只要源队列超长且目标队列短，迁移大概率有益。

M0 则试图**精确估算**远端等待时间（`stale_total / CORES_PER_HOST * (E / cap + T_host)`），但这个估算基于 10us 前的快照。在 MMPP 局部突发条件下：

- 突发期 Burst stay = 500us，即约 50 个同步周期内持续高流量
- 热节点在 10us 内新增任务数可达 `lambda_burst * 10us / 16_hot_nodes ≈` 数十个
- **stale_view 低估热节点队列、高估冷节点队列**

结果：M0 将任务迁移到"以为空闲但实际已拥挤"的节点 → 迁移后实际延迟反而更高 → imr 飙升。

### 3.2 扫描覆盖率的结构性差异

| | B2 | M0 (最终版 #11) |
|---|---|---|
| 每次 CHECK 扫描范围 | **64 个主机**（random start，找第一个 Q>Q_hi） | **1 个主机**（stale_view 最大的那个） |
| 有效覆盖 | 所有热节点均有机会被帮助 | 只帮助 stale_view 最重的 1 个 |

B2 的随机扫描天然具备"分散式救援"能力——突发期间所有 Q>Q_hi 的热节点都会被发现。M0 仅帮助 1 个主机，但热节点有 16 个。

> 指导文档的原始设计中，CHECK_MIGRATION 描述为"遍历**本地**就绪队列"，暗示每个 host 独立运行自己的周期性扫描。这意味着在真实设计中，**所有 64 个 host 同时并行扫描自己的队列**，16 个热节点可以同时各自迁移 1 个任务/us。但我们的集中式模拟器将其简化为全局单次扫描 + 选 1 个 host，严重削弱了 M0 的并行疏导能力。

### 3.3 高负载下的 M0 恶性循环

```
高负载 → 突发时队列变化快 → stale 数据严重偏差
    → 远端估计不准 → 大量无效迁移 (imr ↑)
        → margin 自适应调大 → 有益迁移也被阻挡
            → P99 改善消失 → M0 反而更差
```

在 rho=0.85/0.92 下，这个恶性循环无法打破：
- 不调大 margin → imr 超标 (>0.30)
- 调大 margin → 失去迁移能力，退化为 B1

### 3.4 B2 在局部突发下的天然适配性

B2 的阈值 Q_hi/Q_lo 使用的是 stale_view 的 p25/p75 分位数。当热节点的本地队列**实时**超过 Q_hi 时（这是精确的），就触发迁移。关键在于：

1. **触发条件 100% 精确**：`local_total_queue_len()` 是实时值
2. **选择任务简单可靠**：取最长队列的尾部任务（最新到达，可回收性最高）
3. **不需要远端估计精度**：只需目标 `stale_view[dst] <= Q_lo`，容错空间大

## 五、结论与方法论反思

### 5.1 当前实现的结构性局限

当前 M0 实现存在两个与指导文档设计意图的偏差：

1. **集中式单点扫描 vs 分布式并行扫描**
   - 指导文档："遍历**本地**就绪队列"→ 每个 host 独立、并行扫描
   - 当前实现：全局单事件 → 每 1us 仅扫 1 个 host
   - **影响**：16 个热节点只能逐个被疏导（16us 才轮一遍），而 B2 每次 CHECK 覆盖全部 64 host

2. **迁移收益估算的根本精度问题**
   - M0 的核心优势"精准预测队列延迟"建立在**本地状态完全可观测**的前提上
   - 但**远端状态**只有 10us 前的快照，在 MMPP 突发条件下严重失准
   - 指导文档的相关前提条件是"异构降频 + 突发流量"场景（实验二），其中**降频节点**和**正常节点**的负载差异是稳定的、不受 stale 影响的
   - 当前 W2 场景（同构 + 纯 MMPP 突发）→ 负载差异完全由瞬态突发产生 → 对 stale 敏感度极高

### 5.2 可能的突破方向

| 方向 | 预期效果 | 所需改动 | 风险 |
|------|----------|----------|------|
| **并行化 CHECK_MIGRATION**：每个 host 独立运行 CHECK 事件 | M0 获得与 B2 相当的覆盖率 | 重构事件调度，64 个独立 CHECK 定时器 | 迁移总量可能飙升 |
| **加入异构降频节点**（如指导文档的 15% nodes, C=0.2） | 创造**稳定**负载差异（不依赖 stale），M0 预测更准 | 修改 node 初始化 + 工作负载 | 已不属于 W2 场景（偏向实验二） |
| **提高 SYNC_LOAD 频率**（如 2us 或 5us） | 降低 stale 偏差 | 改一个常量 | 原则上可调，但实际系统中同步开销不可忽视 |
| **M0 仅在突发期发作前迁移**（利用 MMPP 状态切换感知） | 超前行动，在 stale 还准的时候迁移 | 需知道 MMPP 状态 → 可能违反公平 | 实际系统中无法知道突发何时到来 |

### 5.3 核心判断

> **在当前 W2（同构 + MMPP 局部突发）场景下，M0 基于陈旧远端信息的收益估算模型，在中高负载（rho≥0.85）时与 B2 的纯本地阈值判断相比，存在根本性的信息精度劣势。**

这不是参数调优（margin、K_DST、scan 策略）可以解决的，而是 **决策架构层面的不对称**：

- B2 的迁移决策只需要两个信息：「我本地是否过载（精确）」+「远端是否够空（近似足够）」
- M0 的迁移决策需要三个信息：「本地风险（精确）」+「远端等待（不精确）」+「迁移收益 > margin（两个不精确量的差）」

差值估计对误差极为敏感——两个方向性正确但各有几十 us 偶然误差的估计量相减，可以产生正负颠倒的结论。

## 迭代 14：并行化 CHECK_MIGRATION（per-host） ★ 验收通过

### 修改内容

| 项目 | 变更 |
|------|------|
| CHECK_MIGRATION 架构 | **M0**: 从全局单事件 → 64 个 per-host 独立事件（随机偏移打散），每个 host 每 1us 扫描自己的 16 个 cores |
| B2 保持不变 | B2 仍使用集中式单事件 CHECK（随机起始扫描全 64 hosts），保持其原始设计 |
| M0 margin | 从多级自适应 margin（1.5/4.5/9/18us）恢复为固定 1.5us（多级 margin 在 #12-13 证明有害） |
| M0 should_skip_migration | 移除 imr>0.50 跳过机制（在 #12-13 证明导致退化为 B1） |
| M0 预算 | 有效阈值从 5% 降至 4.5%（`M0_BUDGET * 0.90`），吸收 64-host 并行超调 |

### 设计理由

指导文档 CHECK_MIGRATION 描述为"遍历**本地**就绪队列"，意味着每个 host 独立并行扫描。此前的集中式实现每 1us 仅扫 1 个 host（stale_view 最重载），16 个热节点需 16us 才能逐个被覆盖。并行化后，所有热节点可同时自救。

B2 保持集中式原因：B2 的设计是"全局巡检找第一个过载 host"，属于不同的迁移范式，强行并行化反而降低效果（实验证实）。

### 分析

1. **rho=0.50 大幅改善（+65.9%）**：低负载下 stale 偏差小，per-host 并行扫描让每个 host 精准地迁移自己队列中的高风险任务。imr 极低（<0.001），说明迁移决策高质量。
2. **rho=0.85 显著改善（+40.1%）**：关键突破——这是之前 13 轮迭代未能达标的高负载点。per-host CHECK 使 16 个热节点可以**同时**发起迁移（而非排队等待全局轮转），大幅提升了疏导速度。imr=0.17（远低于 0.30 上限），说明即使 stale 不完全准确，per-host 的本地扫描决策质量仍可接受。
3. **rho=0.70 退化（−172%）**：中等负载下，per-host 并行导致 all 64 hosts 竞争迁移预算，非热节点的低收益迁移挤占了热节点的份额。与 #11（集中式扫最重 host）相比，预算分配效率下降。
4. **rho=0.92 退化（−186%）**：极高负载下所有主机均饱和，迁移找不到空闲目标。M0 的 per-host 并行加剧了无效迁移尝试。

### 核心教训

1. **CHECK 扫描架构比参数调优更重要**：13 轮参数调优（margin/K_DST/scan策略/检测阈值）均未突破 1 个 rho 点。第 14 轮的架构变更（集中式 → 分布式并行）直接解锁了第二个 rho 点。
2. **算法的 CHECK 结构应匹配其设计范式**：B2 适合集中式巡检（全局找过载 host），M0 适合 per-host 本地扫描（每个 host 独立评估自己的队列风险）。将 B2 强行并行化反而降低效果。
3. **多级自适应 margin 有害**：在 MMPP 环境下，imr 的全局累积统计无法区分"当前是否在突发期"，导致突发刚开始时 margin 被高值锁住，抑制了最需要迁移的时间窗口。固定 1.5us margin 表现更稳定。
4. **预算管理需考虑并行度**：64-host 并行 CHECK 的有效预算阈值需低于名义 5%（使用 4.5%），否则并行超调导致 mr 越限。
