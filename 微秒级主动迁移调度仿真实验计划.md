# 微秒级主动迁移调度仿真实验计划（已冻结执行版）

## 0. 顶层实验路线与当前算法规划（2026-04-20）

本计划当前服务于论文级算法 `DQB-PM`（Distribution-aware Queue-Batch Proactive Migration）的验证。已有 `M0 Host-ProactiveMigration` 与 `M1 AQB-PM` 结果不废弃，但它们的角色调整为 baseline / 过渡版本：`M0` 用于代表逐任务预测迁移，`M1` 用于代表逐任务候选排序后的批量选择机制，最终目标算法应证明“分布感知队列批迁移”在效果、控制开销和可实现性上更适合微秒级 RPC 场景。

### 0.1 当前论文主线

论文主线固定为：

> 微秒级任务量巨大，逐任务打分和排序无法作为真实控制路径。更合理的做法是根据服务时间分布构建核心队列摘要，用排队论估计一批任务的 SLO 风险，并在 stale load view 下对高风险批次执行有界迁移。

因此，实验不只报告 `P99/P99.9` 是否下降，还必须证明：

- 控制路径复杂度不随队列长度线性增长。
- 双峰和重尾分布下的风险模式能被队列摘要捕捉。
- 批迁移不会制造迁移风暴或目标端拥塞。
- 服务时间估计从 oracle 变为 EWMA/noisy 后仍然有收益。
- c6525-25g 上测得的网络/主机/控制面开销不会推翻仿真结论。

### 0.2 最终算法与基线关系

| 方法 | 实验角色 | 主要回答的问题 |
| :---- | :---- | :---- |
| `B1 Power-of-k` | 初始分发 baseline | 仅靠入口负载均衡是否足够 |
| `B2 ReactiveMigration` | 被动迁移 baseline | 等队列过载后再迁移是否太晚 |
| `M0 ProactiveMigration` | 逐任务预测迁移 baseline | 主动预测机制本身是否有效 |
| `M1 AQB-PM` | 逐任务候选批量选择过渡版本 | 批量决策是否有收益，以及逐任务排序有哪些问题 |
| `DQB-PM` | 最终目标算法 | 分布摘要 + 队列批迁移是否能在低开销下改善尾延迟 |

后续实验报告中应避免把 `M0` 继续称为“最终目标方法”。建议统一使用：

- `M0-TaskPM`：逐任务预测迁移。
- `M1-AQB`：逐任务候选批选择。
- `M2-DQB` 或 `DQB-PM`：分布感知队列批迁移。

### 0.3 DQB-PM 必须验证的算法假设

1. **分布假设**：任务服务时间不是任意噪声，而是存在可利用的分布结构，例如双峰短/长任务、重尾 mice/elephant 任务。
2. **队列风险假设**：P99/P99.9 尾延迟主要来自队列形态，例如 `ShortBehindLong`、`MiceBehindElephant`、慢节点批量积压，而不只是单个任务偶然变慢。
3. **低开销假设**：核心队列摘要可以在常数桶数下维护，控制面不需要全队列逐任务排序。
4. **迁移收益假设**：少量等待请求的跨主机转发可以提前缓解局部队列压力，但只有在目标端未饱和时才有效。
5. **现实可校准假设**：c6525-25g 上测得的 RPC 转发成本、队列摘要成本和控制周期成本可以被纳入仿真，并且不会使算法收益消失。

### 0.4 实验闭环

最终实验必须形成四个层次：

1. **正确性/可重放门禁**：固定 seed、固定 warmup、固定 measurement、统一事件顺序、统一 stale view 约束。
2. **机制验证**：证明 DQB-PM 相比 M0/M1 的收益来自分布摘要、批次风险估计和 incoming reservation，而不是偶然调参。
3. **大规模仿真**：在同构、异构、突发、重尾和饱和边界下验证 P99/P99.9、SLO 违约率、迁移副作用和控制面开销。
4. **小规模实机校准**：在 CloudLab c6525-25g 2-4 台机器上测量关键物理常数，并用这些常数回灌仿真。

### 0.5 前沿研究转化为实验要求

上一次文献调研得到的关键结论必须落到实验设计中，而不是只停留在 Related Work：

- 来自 Horus/RackSched/R2P2 的要求：必须报告 post-dispatch queue repair 与 initial dispatch 的差异。因此实验中必须保留 `B1 Power-of-k`、workload-aware dispatch 或 local priority baseline，证明 DQB-PM 不是普通入口负载均衡。
- 来自 Shinjuku/Concord 的要求：必须展示 HoL blocking 机制。因此实验中必须拆分 short/long 或 mice/elephant 的 SLO violation，而不能只报告混合 P99。
- 来自高速近似调度工作的要求：必须证明不做全量排序。因此实验中必须输出 `control_decision_cost_est_us`、`summary_update_cost_est_us`、候选数和 batch 大小分布。
- 来自 SIRD/Homa 类 receiver-driven 思路的要求：必须验证目标端保护。因此实验中必须做 `no-reservation` 消融，并输出 `dst_incoming_reserved_us` 与 per-destination migration distribution。
- 来自 overload control 工作的要求：必须展示全局饱和时的退化边界。因此实验中必须做 high-rho/saturation 场景，并验证 no-migrate 或 low-batch guard 不会造成比 baseline 更差的迁移风暴。
- 来自 CloudLab 官方平台约束的要求：必须把 c6525-25g 实测成本回灌仿真。因此主结果至少要给一组“初始常数”和一组“CloudLab-calibrated constants”的对比。

## 1. 目标与边界
- 主目标：在相同资源预算和相同陈旧视图约束下，验证 `DQB-PM` 在 `P99/P99.9` 上优于主机间 baseline、逐任务主动迁移 baseline 和当前批选择原型。
- 次目标：同时证明迁移副作用与控制面开销可控，`migration_rate`、`invalid_migration_ratio`、批迁移目标拥塞和队列摘要构建成本维持在可接受范围。
- 研究主问题：主机间等待请求再分配/迁移（L1）。主机内调度（L2）作为执行层与消融组件。最终迁移语义限定为 queued RPC request metadata forwarding，不迁移正在执行状态。
- 控制单位：任务仍是执行与统计单位，但 DQB-PM 的在线控制决策单位必须是 micro-batch / distribution region，而不是单个任务。micro-batch 由 `source_core + service_class + arrival_epoch + risk_region` 定义，批内迁移按 FIFO 前缀或工作量前缀执行。

## 2. 已冻结决策
- 事件模型：`TASK_FINISH > TASK_ARRIVE > TASK_GENERATE`（同时间戳固定优先级）。
- 核心事件：`TASK_GENERATE`、`TASK_ARRIVE`、`TASK_EXECUTE`、`TASK_FINISH`、`SYNC_LOAD`、`CHECK_MIGRATION`。
- 常数与单位：`T_host_us=2.1`、`T_net_oneway_us=3.15`、全链路统一 `us`。
- 服务时间口径：主实验使用 `EWMA`，白盒已知 `E` 作为消融。
- 首轮规模：`64 nodes x 16 cores`。
- 小规模实机：CloudLab `c6525-25g` `2-4` 台。官方硬件口径（https://docs.cloudlab.us/hardware.html）为 AMD EPYC 7302P 16-core 3.00GHz、128GB ECC、双口 Mellanox ConnectX-5 25Gb NIC、两条 25Gb 实验链路；实机实验用于校准 `T_net`、`T_host`、队列摘要构建成本和请求转发成本，不要求完全复现 64-node 大规模仿真的绝对数值。

## 3. SLO 分层协议（已冻结）
- 分层规则：按 `expected_service_time_us` 二分类。
- 短任务：`E <= 20us`；长任务：`E > 20us`。
- 默认 SLO：`SLO_short=40us`、`SLO_long=200us`。
- 敏感性扫描：`(30,150)`、`(40,200)`、`(50,250)`。

## 4. 统计协议（已冻结）
- `warm-up=200k requests`（不计入统计）。
- `measurement window=1M requests`。
- 每配置 `5` 次独立运行。
- seed 固定集合：`{11,23,37,47,59}`。
- 显著性判据：95% CI 基本不重叠且相对提升 `>5%`，并满足至少 `4/5` seeds 方向一致。
- 分位数区间口径：`P99/P99.9` 使用 bootstrap 95% CI（固定重采样次数），全方法统一。

## 5. Baseline 对比协议（主机间）
### 5.1 主集合
- `B0 Host-LocalOnly`：仅入口分发，不跨主机迁移。
- `B1 Host-Power-of-k`：入口 `k` 选最短主机，运行期不迁移。
- `B2 Host-ReactiveMigration`：阈值触发被动迁移。
- `M0 Host-ProactiveMigration`：逐任务预测型主动迁移 baseline。
- `M1 AQB-PM`：逐任务候选排序后的批量选择 baseline。
- `DQB-PM`：分布感知队列批迁移（目标方法）。
- `U0 Ideal-Oracle`：可选上界，仅用于差距参考，不作为主显著性对比。

### 5.2 公平性硬约束
- 所有方法同拓扑、同输入、同常数、同测量窗口、同 seed 集合。
- `B0/B1/B2/M0` 仅可读取同一份 `SYNC_LOAD` 陈旧全局视图。
- 允许读取本机实时状态；禁止读取跨主机实时状态。
- 跨主机迁移统一计入 `T_net_oneway_us + T_host_us`。
- 调参预算一致：每法相同配置点上限，主结果同时报告“统一默认点”和“同预算最优点”。

## 6. B2 规则（已冻结）
- 触发条件：`Q_src >= Q_hi` 或存在目标满足 `Q_dst <= Q_lo` 且有空闲核心。
- 阈值定义：`Q_hi=p75(Q_host)`、`Q_lo=p25(Q_host)`，按预热窗口统计后冻结。
- 候选任务：仅迁移等待队列任务，默认每次 `n_move=1`，源队列尾部选择。
- 目标选择：基于陈旧视图，默认 `k_dst=2` 候选最小队列主机。
- 保护机制：`T_cooldown_us=2`、`H_q=2`、`r_bad=0.3`、迁移预算上限 5%。

## 7. M0 规则（已冻结）
- 触发器：`CHECK_MIGRATION` 周期扫描，`T_check_us=1.0`。
- 扫描对象：仅等待队列，不迁移正在执行任务。
- 三重判定（全部满足才迁移）：
	- 风险约束：`T_elapsed + W_i + E_i/C_src > alpha * SLO_i`
	- 收益约束：`local_total_i > remote_total_i`
	- 防抖约束：`local_total_i > remote_total_i + T_margin_us`
- 默认参数：`alpha=0.8`、`T_margin_us=1.5`、`k_dst=2`、`n_move_per_check=1`。
- 风暴保护：与 B2 同级预算与冷却，窗口 `invalid_migration_ratio > 0.3` 时临时上调 `T_margin_us`。

### 7.1 迁移副作用可接受阈值（已冻结）
- `invalid_migration_ratio <= 0.30`（窗口统计口径与 B2/M0 一致）。
- `migration_rate <= 0.05`（系统级窗口上限）。
- `migration_work_rate` 必须同步报告，避免低任务数迁移率掩盖高工作量迁移。
- 若任一阈值持续超限，则该配置标记为“副作用不可控”，不纳入主结论。

## 7.2 最终目标算法：DQB-PM 分布感知队列批迁移
- 定位：`M0` 保留为逐任务预测迁移基线；最终投稿算法应命名为 `DQB-PM` 或类似名称，核心是基于任务分布的队列批处理，而不是逐任务打分排序。
- 队列摘要：每个核心维护固定大小的服务时间分布桶。双峰负载维护 `short/long` 桶；重尾负载维护 `mice/medium/elephant` 桶。每个桶记录数量、累计工作量、最老任务等待时间、最紧 SLO slack。
- 微批次定义：每个核心按服务时间类别和到达时间窗口形成 micro-batch。一个 micro-batch 关闭后不再加入新请求；若需要迁移，只能迁移该 batch 的 FIFO 前缀或工作量前缀，不允许逐任务打分挑选。
- 批次风险估计：结合当前运行任务剩余时间、队列前缀累计工作量、核心容量 `C_src` 和桶内平均服务时间，估计某一批任务若留在本地是否会超过 `alpha * SLO`。
- 典型风险模式：
	- `ShortBehindLong`：双峰负载下长任务阻塞短任务批。
	- `MiceBehindElephant`：重尾负载下 elephant task 造成小任务批尾延迟恶化。
	- `SlowNodeBatchPressure`：异构慢节点上队列工作量累积导致一批任务共同逼近 SLO。
- 目标选择：基于陈旧 workload 视图、目标容量、短窗口 `incoming_reservation` 和迁移预算选择目标，避免多个源主机同时涌入同一目标。
- 远端可完成性：目标必须满足 `estimated_remote_finish_time < batch_deadline`、`estimated_gain > margin`、`target_harm` 未超阈值。没有可救回该 batch 的目标时不迁移。
- 迁移语义：仅迁移等待队列中的 RPC 请求元数据，不迁移正在执行任务、线程状态或内存页。
- 复杂度要求：每个核心每个控制周期产生 `O(1)` 个批候选；批候选数量与队列长度解耦。允许在维护摘要时扫描固定前缀，但主控制路径不得依赖全队列逐任务排序。
- 风险与验证：必须报告服务时间估计误差敏感性、批大小敏感性、`incoming_reservation` 消融、控制面开销估计，以及短任务/长任务分组 SLO 违约率。

### 7.3 DQB-PM 消融与诊断字段
- 必做消融：
	- `DQB-PM/full`：完整算法。
	- `DQB-PM/no-reservation`：移除目标端 incoming reservation。
	- `DQB-PM/no-saturation-guard`：移除全局饱和保护。
	- `DQB-PM/single-move`：保留分布摘要，但每次只迁移 1 个任务。
	- `DQB-PM/queue-length-only`：不用 workload 和分布桶，仅用队列长度估计。
	- `M1-AQB/task-score`：当前逐任务候选排序版本。
- 必做诊断字段：
	- `batch_type`：`ShortBehindLong`、`MiceBehindElephant`、`SlowNodeBatchPressure`、`GenericPressure`。
	- `micro_batch_count`、`risk_batch_count`、`move_batch_count`。
	- `batch_candidate_count`、`batch_selected_count`、`batch_move_count`。
	- `src_capacity_class`、`dst_capacity_class`。
	- `src_queue_work_us`、`dst_stale_work_us`、`dst_incoming_reserved_us`。
	- `batch_oldest_age_us`、`batch_min_slack_us`、`batch_estimate_confidence`。
	- `migration_work_rate`、`target_harm_est_us`、`per_dst_migrated_work_us`。
	- `short_slo_violation_rate`、`long_slo_violation_rate`。
	- `control_decision_cost_est_us`、`summary_update_cost_est_us`。
- 验收标准：若 DQB-PM 的 P99/P99.9 改善来自明显更高迁移率、不可控 invalid ratio 或目标端拥塞，则不能作为主结论；必须回到算法设计阶段调整 batch cap、reservation window 或 no-migrate guard。

### 7.4 真实性与鲁棒性必测项
- 可迁移比例：
	- `eligible_fraction={1.0,0.75,0.50}`。
	- 目的：验证并非所有请求都可迁移时，算法是否仍能改善尾延迟。
- 服务时间估计：
	- `Oracle-E` 仅作为上界。
	- `EWMA-E` 作为主实验口径。
	- `Noisy-E` 用于测试估计误差放大 batch 迁移风险。
- 控制周期：
	- 固定周期、随机 offset 周期、事件触发 + 周期触发三种方式对比。
	- 目的：避免所有 host 同时检查造成控制脉冲和羊群效应。
- 目标端粒度：
	- host-average target selection。
	- core-aware target selection（增强版）。
	- 目的：评估仅用 host 平均 workload 是否低估目标端排队风险。
- 目标端伤害：
	- 输出迁入 batch 对目标端 short queue 或高优先级 queue 的预估影响。
	- 若 target harm 超阈值，该目标必须被排除。
- 不迁移条件：
	- 必须统计 no-migrate 原因：`no_slack`、`low_gain`、`reservation_full`、`saturated_cluster`、`low_confidence`、`target_harm_high`、`budget_exhausted`。
	- 目的：证明算法不是贪心迁移，而是有明确拒绝条件。
- 在线信息边界：
	- 实验日志必须标明算法决策只使用本地实时摘要、远端陈旧摘要、incoming reservation 和历史服务估计。
	- 禁止在算法路径中使用真实完成时间、未来到达或远端实时队列。

### 7.5 迁移执行模型：批次级决策与传输层对比
- 专家判断：DQB-PM 的主线应采用 **批次级控制决策 + MVP 逐请求转发**。也就是说，算法一次选择一个 micro-batch 或其 FIFO 前缀，但仿真事件层可以把批内请求逐个投递到远端。这样不依赖未验证的传输层打包收益，更适合作为 INFOCOM MVP。
- 论文表述边界：
	- MVP 可以称为 `batch-level decision` 或 `batch-selected migration`。
	- MVP 不应声称已经实现 `transport-level batching` 或网络头部摊销。
	- Plus 版本只有在 c6525-25g microbenchmark 校准后，才可以报告真实 batch forwarding 收益。
- 必做实验变体：
	- `DQB-PM/individual-forwarding`：默认 MVP。批次统一选择目标，但批内请求分别产生远端到达事件。
	- `DQB-PM/single-move`：保留分布摘要，每轮只迁移一个请求，用于证明“批次级迁移”不是可有可无。
	- `DQB-PM/batch-forwarding`：Plus 版本。同一 `(src,dst)` 的请求描述符合并为一个 batch arrival，使用校准后的批传输成本模型。
- Plus 成本模型：
	- `T_batch(k)=T_net_base+T_batch_header+k*T_request_descriptor+T_batch_unpack`。
	- 需要记录 `batch_size`、`batch_header_cost_us`、`per_descriptor_cost_us`、`batch_unpack_cost_us` 和 `batch_forwarding_gain_us`。
- 评价指标：
	- `P99/P99.9`、`short/long SLO violation`、`migration_count_rate`、`migration_work_rate`。
	- 控制面消息数、迁移字节数、平均 batch 大小、目标端排队伤害。
	- `individual-forwarding` 与 `batch-forwarding` 的差异必须单独报告，不能混入主算法收益。
- 关键约束：
	- 不允许为了攒大批次而额外等待请求。
	- batch 必须来自已经存在的核心等待队列和风险区域。
	- 若 batch forwarding 没有明显收益，主结论仍应回到“分布感知批次级控制降低决策开销并改善尾延迟”。

## 8. 负载协议（已冻结）
- `W1` 主结果：Poisson + Bimodal，服务时间 `80%=5us, 20%=100us`。
- `W2` 突发鲁棒性：MMPP(2-state) + Bimodal，`lambda_burst=1.5*lambda_normal`，停留均值 `Normal=5000us, Burst=500us`。
- `W3` 重尾泛化：Poisson + Lognormal，均值对齐 `E[S]=24us`，初值 `sigma=1.0`，`mu=ln(24)-0.5*sigma^2≈2.678`。
- 扫描策略：`W1` 全扫描 `rho=0.10..0.95, step=0.05`；`W2/W3` 代表点 `rho={0.50,0.70,0.85,0.92}`。

### 8.1 负载强度定义（rho -> lambda，已冻结）
- 定义：`rho = lambda_global * E[S] / (num_hosts * cores_per_host)`。
- 反解：`lambda_global = rho * (num_hosts * cores_per_host) / E[S]`。
- 当前规模：`num_hosts=64`，`cores_per_host=16`，总核心 `1024`。
- W1 示例：`E[S]=24us` 时，`lambda_global = rho * 1024 / 24`（requests/us）。
- 说明：`E[S]` 仅指服务时间分布，不含迁移网络惩罚与主机处理惩罚项。

### 8.2 W2 MMPP 生成口径（已冻结）
- 两状态：`Normal` 与 `Burst`。
- 到达率：`lambda_burst = 1.5 * lambda_normal`。
- 平均停留：`Normal=5000us`，`Burst=500us`（等价为指数停留时间参数化实现）。
- 初始状态：`Normal`。
- 实现约束：使用固定 seed 驱动状态转移与到达生成，保证可重放。

## 9. 执行流程（只保留步骤与验收）

## 13. 风险与备选方案

- 若 DQB-PM 效果不如 M1：将主张调整为“以相近尾延迟换取显著更低控制开销”，并在控制面开销模型中比较两者。
- 若服务时间估计误差导致收益消失：降低对任务类型精确分类的依赖，改为更粗的 `mice/elephant` 桶和更保守的 margin。
- 若 incoming reservation 不能缓解目标拥塞：加入 per-destination 短窗口迁移上限，并把目标端拥塞作为失败案例讨论。
- 若 CloudLab 实测延迟高于初始常数：用实测常数重跑仿真，强调算法在较高迁移成本下的 no-migrate guard 和适用边界。
- 若所有场景收益都不稳定：转向“诊断型论文”风险较高，不建议硬投；应收敛到最稳定的 workload class，例如异构 + burst + 双峰短任务保护。
